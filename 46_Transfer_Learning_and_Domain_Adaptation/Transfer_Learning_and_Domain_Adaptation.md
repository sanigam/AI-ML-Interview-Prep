# Transfer Learning and Domain Adaptation

📺 **Video Lecture:** https://youtu.be/TqFEGBhRZbk

## Interview Anchor
- **Transfer Learning Paradigm:** Pre-training on large datasets then fine-tuning on target tasks to leverage learned representations
- **Domain Adaptation Problem:** Handling distribution shifts between source (training) and target (deployment) domains
- **Few-Shot and Zero-Shot Learning:** Generalizing to new tasks and classes with minimal or no target data

## Key Concepts Overview
Transfer learning has revolutionized deep learning by enabling models to leverage knowledge from large pre-trained datasets (ImageNet, BERT, GPT) and apply it to downstream tasks with limited data. The key insight is that early layers learn general features (edges, shapes) while later layers learn task-specific features, so fine-tuning later layers can adapt a model to new domains efficiently. Domain adaptation extends this by explicitly addressing distribution shifts—when training and deployment data differ—a critical problem in real-world ML. Interviewers assess your understanding of when transfer learning helps, how to fine-tune effectively, and how to diagnose and fix negative transfer. Strong transfer learning knowledge demonstrates that you can build practical ML systems that don't require billions of labeled examples.

### Q1: What is transfer learning and why does it work? Explain the intuition.

**A:** Transfer learning reuses knowledge learned on one task to improve performance on another. For vision, a CNN trained on ImageNet learns edge detectors in early layers, textures in middle layers, and object-specific patterns in late layers; fine-tuning on a new task (e.g., medical imaging) reuses the low-level feature detectors because edges and textures are universal across domains. For NLP, BERT pre-trains on 3.3B words using masked language modeling, learning contextual word representations; fine-tuning on sentiment classification (a downstream task) initializes with these representations rather than random weights, converging faster and generalizing better. The mechanism: gradient descent initializes the network with useful weights (from pre-training) rather than random, reducing the optimization landscape to a region where good solutions exist. This is especially powerful when the target dataset is small (1k-100k examples)—training from scratch requires millions of examples to learn good features, but transfer learning achieves similar performance with far fewer examples. Transfer learning also improves robustness; models trained on diverse pre-training data generalize better to distribution shifts. The limitation: transfer learning works when source and target tasks share similar low-level features; it may hurt if domains differ drastically (negative transfer).

### Q2: Describe the feature extraction vs. fine-tuning strategies. When would you use each?

**A:** Feature extraction freezes pre-trained weights and trains only the task-specific head: load BERT, freeze all transformer layers, train a linear classifier on top. This is fast (only train the classification head, one epoch on target data) and stable (frozen weights won't overfit on small target datasets). Use feature extraction when: (1) target dataset is very small (<1k examples), (2) target domain is very different from pre-training domain, (3) compute is limited (no GPU for full fine-tuning). Fine-tuning updates all weights, starting from pre-trained values: load BERT, train all 12 transformer layers plus the classification head. This is slower but more flexible—the model adapts its learned features to the target domain. Fine-tuning performs better when: (1) target dataset is large (>50k examples), (2) target domain is similar to pre-training, (3) you have sufficient compute. A hybrid approach, gradual unfreezing, trains the head first, then thaws earlier layers progressively: epoch 1 train head only, epoch 2 fine-tune last transformer layer + head, epoch 3 fine-tune last 2 layers, etc. This balances stability and adaptation, performing well across dataset sizes. In practice, start with feature extraction (quick baseline), then fine-tune if performance plateaus and data permits. Some practitioners use discriminative fine-tuning (lower learning rates for early layers, higher for later): `learning_rates = [1e-5, 1e-4, 1e-3]` for layers in order, accounting for the intuition that early layers need less updating.

### Q3: How does layer freezing work? Walk through a training example.

**A:** Layer freezing sets `layer.requires_grad = False` in PyTorch, preventing gradient computation and weight updates for that layer. Example: `model = torchvision.models.resnet50(pretrained=True); for param in model.layer1.parameters(): param.requires_grad = False; optimizer = optim.Adam(filter(lambda p: p.requires_grad, model.parameters()), lr=0.001)` freezes layer1 (early layers), only updates layer2-4 and the classification head. During backprop, gradients still flow through frozen layers (for activation computation), but `optimizer.step()` skips updating their weights because `.requires_grad = False`. This saves memory and compute—fewer parameter gradients to store and update. Frozen layers act as fixed feature extractors; the trainable layers learn task-specific mappings. For progressive unfreezing: `def unfreeze_layer(layer): for param in layer.parameters(): param.requires_grad = True` then call `unfreeze_layer(model.layer2)` after a few epochs. The learning rate schedule matters; unfreezing layers initialized with random weights requires careful tuning to prevent catastrophic forgetting (fine-tuning erases pre-trained knowledge). In code, discriminative fine-tuning sets different learning rates: `param_groups = [{'params': model.layer1.parameters(), 'lr': 1e-5}, {'params': model.classifier.parameters(), 'lr': 1e-3}]` gives early layers tiny updates while task-specific layers train faster.

### Q4: What is domain adaptation and how does it differ from transfer learning?

**A:** Domain adaptation addresses a key limitation of transfer learning: distribution shift between source (training) and target (test) data. Transfer learning assumes source and target are from the same distribution; domain adaptation explicitly handles cases where they differ. For example, training an object detector on synthetic images (source) then deploying on real images (target)—the domains shift (rendering artifacts, lighting, textures). Covariate shift means P(X_target) differs from P(X_source) but P(Y|X) is unchanged (marginal X distribution changes, conditional stays same). Dataset shift (broader) includes any P(X, Y) difference. Concept drift occurs when P(Y|X) changes over time (e.g., user preferences evolve). Domain adaptation strategies: (1) unsupervised DA—adapt to target without labels, (2) semi-supervised DA—use few target labels, (3) self-training—train on target pseudo-labels. The key insight: if you ignore distribution shift and train normally, the model overfits to source domain quirks and fails on target. Domain adaptation methods explicitly minimize domain discrepancy through adversarial training, maximum mean discrepancy (MMD), or self-supervised learning on target data. Unlike transfer learning (assumes shared task), domain adaptation assumes the task is the same but data distribution differs, requiring different technical approaches.

### Q5: Explain covariate shift, concept drift, and dataset shift. Provide examples.

**A:** Covariate shift occurs when P(X) changes but P(Y|X) remains constant. Example: training on historical user data (feature distribution), deploying on new users with different demographics—the features shift, but the relationship between features and churn probability stays constant. Covariate shift is relatively benign; if you reweight training samples by P_target(x) / P_source(x), standard ML recovers unbiased estimates. Concept drift occurs when P(Y|X) changes—the decision boundary shifts over time. Example: training on 2020 economic data to predict loan default, deploying during a 2024 recession—unemployment affects default probability differently, so the relationship P(default|features) has drifted. Concept drift is harder to fix; requires retraining on recent data or online learning. Dataset shift (broadest) encompasses both. Example of pure dataset shift: training on English text to predict sentiment, testing on text translated from Chinese—both P(X) and P(Y|X) change because translation artifacts and cultural differences affect sentiment signals. For covariate shift, domain adaptation reweights or aligns feature distributions using adversarial methods or MMD. For concept drift, you must retrain on recent data; there's no getting around the fact that the underlying relationship changed. Practitioners monitor performance degradation to detect drift; if accuracy drops >5% month-over-month, retraining is often triggered. Understanding these distinctions is crucial for diagnosing why a deployed model fails.

### Q6: What is domain adversarial training (DANN)? Walk through the algorithm.

**A:** Domain adversarial neural networks train an encoder to learn features that are both task-discriminative (good for the target task) and domain-invariant (indistinguishable between source and target). Architecture: encoder (shared), task classifier head, and domain classifier. Loss = task_loss + lambda * domain_loss, where domain_loss is a binary classification loss distinguishing source vs. target. Algorithm: (1) forward pass—compute task predictions and domain predictions, (2) compute task loss (e.g., cross-entropy on source labels), (3) compute domain loss (binary cross-entropy on source/target domain labels), (4) backprop: gradients flow normally to the task head, but through a gradient reversal layer (GRL) to the domain classifier—GRL multiplies by -lambda, making the encoder optimize to fool the domain classifier. The intuition: the task classifier wants to classify correctly; the domain classifier wants to identify source vs. target. The encoder is caught in a min-max game: it learns features that are good for the task while confusing the domain classifier. At equilibrium, the encoder produces features where source and target are indistinguishable, but task performance remains high. In practice, gradually increase lambda during training; start with small lambda (task learning dominates), then increase to enforce domain invariance. Implementation: in PyTorch, use a GradientReversal layer with custom backward(). DANN works well empirically—e.g., adapting from synthetic to real objects or adapting across visual domains. A limitation: DANN assumes tasks are identical across domains; if P(Y|X) differs (concept drift), DANN won't help because fooling the domain classifier doesn't fix task-level drift.

### Q7: What is maximum mean discrepancy (MMD) and how does it address domain adaptation?

**A:** MMD measures the distance between two distributions by comparing their feature means in a high-dimensional space. Formula: MMD^2(P, Q) = || E_P[phi(X)] - E_Q[phi(X)] ||^2, where phi is a feature mapping. For distributions far apart in original space, embedding via phi (e.g., nonlinear kernel map) can reveal differences. Empirically: MMD(P, Q) = || (1/n) sum_i phi(x_i) - (1/m) sum_j phi(y_j) ||^2 computes kernel means. For domain adaptation: add a regularizer to the task loss that penalizes MMD between source and target feature distributions: Loss = task_loss + lambda * MMD(source_features, target_features). This forces the model to learn features where source and target align, reducing domain shift. MMD is differentiable (computable on mini-batches), so it integrates smoothly into neural network training. Advantages over DANN: simpler (no adversarial min-max game), more stable, easier to implement. Disadvantages: assumes MMD is the right metric (empirically often is, but not always), and MMD can be expensive for large feature dimensions (kernel computation). In practice, use a kernel-based MMD: `sum_{i,j} K(x_i, x_j) - 2 * sum_{i,j} K(x_i, y_j) + sum_{i,j} K(y_i, y_j)` with RBF kernel K. MMD works best for covariate shift; like DANN, it can't fix concept drift.

### Q8: What is few-shot learning? Explain prototypical networks.

**A:** Few-shot learning generalizes to new classes with minimal examples (e.g., 1-shot, 5-shot). Prototypical networks learn a metric space where examples of the same class cluster together. Algorithm: (1) during training, sample N-way K-shot episodes (N classes, K examples per class), (2) compute prototype (mean embedding) for each class: prototype_c = (1/K) sum_k embed(x_ck), (3) predict new query point's class by nearest prototype: argmin_c || embed(query) - prototype_c ||_2. Loss during training: softmax cross-entropy over class distances. Key insight: prototypical networks learn an embedding such that within-class distances are small and between-class distances are large. The same metric space generalizes to new classes—at test time, you see 5 new classes with 5 examples each, compute their prototypes, and classify queries by nearest prototype. Contrast this to fine-tuning: fine-tuning updates all model weights on the new task (requires compute and can overfit on few examples), while prototypical networks use the pre-trained embedding (one forward pass per prototype). Prototypical networks work well empirically for vision (5-way 5-shot classification achieves 99%+ on miniImageNet) and NLP (few-shot sentiment classification). Limitations: assumes classes can be separated in a fixed metric space (Euclidean distance); heterogeneous classes may require learned distance metrics (Siamese networks). The name comes from prototype theory in psychology—humans recognize objects by comparing to mental prototypes, not explicit feature lists.

### Q9: What is MAML (Model-Agnostic Meta-Learning)? How does it differ from prototypical networks?

**A:** MAML learns model parameters that are good initialization points for few-shot adaptation. Instead of learning a fixed metric space, MAML learns weights such that a few gradient steps on a new task converge quickly. Algorithm: (1) sample tasks from distribution, (2) for each task, (3a) compute inner loop loss on K support examples: L_support = loss(model, support_data), (3b) take one or few SGD steps: theta' = theta - alpha * grad(L_support), (3c) compute outer loop loss on query examples: L_query = loss(theta', query_data), (4) update meta-parameters: theta -= beta * grad(L_query). The outer loop optimizes for fast task adaptation, not final performance—you're learning initial weights that converge quickly on new tasks. At test time, take K examples from a new class, update weights once: theta_new = theta - alpha * grad(loss(K_examples)), then predict with theta_new. Key differences from prototypical networks: prototypical networks are task-agnostic (apply to any classification problem with the learned metric), while MAML is model-agnostic (works with any differentiable model: CNNs, RNNs, etc.). MAML is more computationally expensive (inner loop SGD during training), but often achieves better few-shot performance because it optimizes for the actual test-time update process. In practice, MAML is stronger empirically (few-shot classification, few-shot RL) but harder to implement; prototypical networks are simpler and still effective. Recent variants: first-order MAML (FOMAML) skips second-order Hessian computation, reducing memory; bilevel optimization interprets MAML as a bilevel optimization problem.

### Q10: What is zero-shot learning? How does it differ from few-shot learning?

**A:** Zero-shot learning classifies on classes never seen during training, using auxiliary information like class descriptions or semantic embeddings. Example: you train a classifier on "dog" and "cat" with descriptions ("four-legged animal", "carnivore"). At test time, given "tiger" description, the model infers "tiger" has similar semantic properties to "cat" and classifies accordingly, without seeing a single tiger image. Method 1 (semantic embeddings): map images and class descriptions to a shared embedding space, classify by nearest semantic embedding. `image_embedding = encoder(image); class_embedding = text_encoder(description); logits = image_embedding @ class_embedding.T; prediction = argmax(logits)`. Method 2 (attribute-based): describe classes by attributes (four-legged, furry, carnivore) and train a classifier to predict attributes, then map attributes to classes. Difference from few-shot: few-shot uses a few examples of the new class (5 images of tiger), while zero-shot uses no examples, only descriptions or attributes. Zero-shot is harder (no direct supervision on the class) but more practical (handles novel classes without retraining). A limitation: zero-shot requires manual descriptions or attribute labels (effort-intensive), and performance degrades if semantic information doesn't transfer (e.g., similar description but visually distinct classes). Modern approaches use large language models: CLIP encodes images and text (including class names) in a shared space, enabling zero-shot classification by comparing image embeddings to class name embeddings. CLIP pre-trained on 400M image-text pairs achieves impressive zero-shot transfer without fine-tuning—a paradigm shift showing that scale and diverse data enable emergent zero-shot abilities.

### Q11: What is multi-task learning? Why does it help generalization?

**A:** Multi-task learning trains a single model on multiple related tasks simultaneously, sharing representations to improve generalization. Architecture: shared encoder, multiple task-specific heads (classifiers or regressors). Loss = loss_task1 + loss_task2 + loss_task3. Example: predict user demographics (age, gender, location) and next purchase category from a single user behavior sequence—the shared encoder learns features useful for all tasks, and task-specific heads specialize. Why it helps: (1) regularization—if one task has limited data (age prediction), its gradient alone could overfit; other tasks' gradients regularize (similar to dropout), (2) feature learning—different tasks supervise different feature aspects; jointly optimizing creates richer representations, (3) inductive bias—sharing structure encodes the assumption that tasks are related, biasing the model toward reusable features. Contrast: multi-task learning assumes tasks share representations; if tasks are unrelated, sharing hurts (negative transfer). Handling imbalanced task datasets: if task1 has 1M examples and task2 has 1k, task1 dominates; use task weighting (`loss = w1 * loss1 + w2 * loss2`, tune weights via validation) or batch sampling (oversample task2 examples). Uncertainty weighting is an elegant approach: learn task uncertainty sigma during training, weight loss by 1/sigma^2—the model automatically downweights high-uncertainty (unreliable) tasks. Multi-task learning is widely used in industry: YouTube's recommendation system predicts clicks, dwell time, and shares jointly; language models use next-token prediction + masked language modeling + sentence ordering simultaneously. The limitation: multi-task learning requires curating related tasks; naive combinations can hurt.

### Q12: What is catastrophic forgetting and how does continual learning address it?

**A:** Catastrophic forgetting (interference) occurs when fine-tuning on a new task degrades performance on previous tasks. Example: train a model on task1 (classifying cats vs. dogs), fine-tune on task2 (classifying birds vs. fish); the weights optimized for task2 overwrite weights useful for task1, causing cat/dog accuracy to drop to random. This is a central challenge in continual learning—learning a sequence of tasks without forgetting. Solution 1 (regularization): add a penalty term encouraging weights to stay close to pre-task1 values. `loss = task2_loss + lambda * (theta - theta_old) ^ 2` (elastic weight consolidation; more sophisticated: only penalize important weights identified by task1's Fisher information). Solution 2 (replay): store examples from task1, mix them into task2 training. `batch_task2 = sample(task2_examples, size=k); batch_task1_replay = sample(task1_memory, size=k); loss = loss(batch_task2) + loss(batch_task1_replay)`. Solution 3 (architecture): add task-specific modules (adapters) for each task, sharing a base model. Task1 uses base + adapter1; task2 uses base + adapter2; no weight conflict. Solution 4 (parameter isolation): identify parameters unique to each task (lottery ticket hypothesis), training on non-overlapping subnetworks. Solution 5 (knowledge distillation): train task2 model to mimic task1 model on task1 data, preserving task1 knowledge while learning task2. The limitation: continual learning is fundamentally hard (can't perfectly learn all tasks with finite capacity); trades off performance between tasks. No perfect solution; choose method based on constraints: regularization is cheap but approximate; replay requires memory; architecture approaches scale with task count.

### Q13: How do you fine-tune BERT for downstream NLP tasks? What are best practices?

**A:** BERT pre-trains on 3.3B words via masked language modeling (MLM) and next sentence prediction (NSP), learning contextual representations. Fine-tuning for classification: take the [CLS] token's representation (special token prepended to all sequences), pass through a task-specific linear layer, and fine-tune all BERT weights. Architecture: `input = [CLS] text [SEP]; embeddings = bert_embeddings(input); outputs = bert_encoder(embeddings); cls_output = outputs[0]; logits = classifier(cls_output); loss = cross_entropy(logits, label)`. Best practices: (1) lower learning rate than pre-training (2e-5 to 5e-5 typical) because BERT weights are already optimized; higher rates corrupt learned representations, (2) short warmup (10% of training steps) then linear decay to 0, (3) few epochs (2-5) to avoid overfitting; BERT saturates quickly due to strong pre-training, (4) use Adam with weight decay (L2 regularization parameter 0.01), (5) batch size 8-32 (smaller than pre-training's 256 because target task data is limited), (6) gradient accumulation if memory-limited: accumulate gradients over k mini-batches, update weights once. Example code: `optimizer = AdamW(model.parameters(), lr=2e-5, weight_decay=0.01); scheduler = get_linear_schedule_with_warmup(optimizer, num_warmup_steps=500, num_training_steps=10000)`. For large datasets (>100k examples), standard fine-tuning works; for small datasets (<5k), add layer-wise discriminative fine-tuning (lower learning rates for early layers) or use gradual unfreezing (train head first, then unfreeze layers). A common mistake: using the default Adam learning rate (1e-3), which is too high and destroys BERT's pre-trained knowledge; practitioners often learn this painfully.

### Q14: How does ImageNet pre-training transfer to downstream vision tasks? What's the advantage over training from scratch?

**A:** ImageNet pre-training trains a CNN on 1.2M images across 1000 object classes using supervised learning. Transfer to downstream tasks (medical imaging, autonomous driving, satellite imagery) leverages low-level features learned by ImageNet. Empirical results: ResNet-50 trained from scratch on ImageNet requires 90 epochs and 8 GPUs for 72-hour training; fine-tuned on a downstream task (e.g., 10k medical images) in 2 hours reaches higher accuracy than training from scratch on the same 10k images for 10 hours. The advantage: early CNN layers learn universal features (edges, textures, simple shapes) useful across domains; ImageNet provides this free via pre-training. When transferring: freeze early layers (conv1, layer1, layer2), fine-tune later layers (layer3, layer4, classifier). Layer-wise feature visualization shows that unfrozen early layers overfit to downstream data despite pre-training. For very similar domains (e.g., natural images to fine-grained birds), even the classifier head transfers; for very different domains (natural images to X-rays), earlier layers transfer better—medical images have different texture/scale distributions. A complication: ImageNet pre-training is biased toward object recognition—models exploit high-frequency textures and ignore shapes (texture bias). For tasks requiring shape understanding, texture-debiased ImageNet pre-training or domain-specific pre-training (e.g., RadImageNet for medical imaging) works better. For very small downstream datasets (<1k examples), feature extraction (freeze all but the head) prevents overfitting. The lesson: domain-specific pre-training > ImageNet pre-training > training from scratch for small-data transfer scenarios.

### Q15: When and why does transfer learning fail (negative transfer)? How do you diagnose it?

**A:** Negative transfer occurs when fine-tuning on a target task hurts performance compared to training from scratch. Causes: (1) domain mismatch—source and target are too different (pre-trained knowledge doesn't apply; e.g., training vision model on ImageNet, transferring to microscopy images), (2) task mismatch—pre-training task is unrelated to target (pre-training on classification, transferring to object detection without adaptation), (3) distribution shift in deployment—model overfits to source domain quirks that don't generalize (pre-training includes dataset artifacts that confuse deployment), (4) insufficient target data—pre-training biases dominate, preventing the model from adapting to target (e.g., fine-tuning on 10 images hurts because pre-training's inductive bias is too strong). Diagnosis: compare fine-tuned model vs. training from scratch on the target task, measuring both test performance and learning curves. If fine-tuned model is slower to converge or saturates lower, negative transfer is likely. Attribution: check intermediate representations—if pre-trained features are unrelated to target (cosine similarity of embeddings << 1), domain mismatch is the issue; if pre-trained features are close to final solution but fine-tuning diverges, task mismatch or insufficient data is likely. Fixes: (1) use feature extraction instead of fine-tuning (preserve pre-trained knowledge while training task-specific head), (2) lower fine-tuning learning rate and fewer epochs (reduce overfitting to target), (3) use domain adaptation methods (DANN, MMD) to align representations, (4) collect more target data if possible, (5) try different pre-training datasets or models (CLIP, domain-specific pre-training). In practice, pre-trained models rarely perform worse than training from scratch when fine-tuning is done carefully (proper learning rates, layer freezing); most negative transfer comes from careless tuning (too-high learning rate, wrong layer selection).

## Interview Cheatsheet

**Key Terms:**

- **Transfer Learning:** Reusing knowledge from pre-training on large data to improve performance on downstream tasks
- **Feature Extraction:** Freezing pre-trained weights, training only the task-specific head
- **Fine-Tuning:** Updating all weights starting from pre-trained values
- **Layer Freezing:** Setting `requires_grad = False` to prevent weight updates during training
- **Covariate Shift:** P(X) changes between source and target, but P(Y|X) stays constant
- **Concept Drift:** P(Y|X) changes over time; decision boundary shifts
- **Dataset Shift:** Any distribution difference between source and target
- **Domain Adversarial Training (DANN):** Learning domain-invariant features via gradient reversal
- **Maximum Mean Discrepancy (MMD):** Distance between distributions in feature space
- **Few-Shot Learning:** Generalizing to new classes with minimal examples (1-5 per class)
- **Zero-Shot Learning:** Classifying on unseen classes using semantic descriptions or attributes
- **Prototypical Networks:** Learning metric space where examples cluster by class
- **MAML:** Learning initial weights that adapt quickly to new tasks
- **Multi-Task Learning:** Training on related tasks simultaneously to share representations
- **Catastrophic Forgetting:** Degradation on old tasks when fine-tuning on new tasks
- **Continual Learning:** Learning sequences of tasks without forgetting previous knowledge
- **Negative Transfer:** Fine-tuning hurts performance compared to training from scratch

**Rapid-Fire Q&A:**

- **Q:** Why does ImageNet pre-training help on medical imaging? **A:** Early CNN layers learn universal features (edges, textures); ImageNet provides these at scale, reducing target task data needed.
- **Q:** How do you handle domain shift in deployment? **A:** Monitor performance; if accuracy drops, use domain adaptation (DANN, MMD) or retrain on recent data.
- **Q:** What's the difference between few-shot and zero-shot learning? **A:** Few-shot: given examples of new classes; zero-shot: given descriptions/attributes of new classes only.
- **Q:** Why does MAML work better than prototypical networks empirically? **A:** MAML optimizes for the actual test-time update process; prototypical networks learn a fixed metric space.
- **Q:** How do you prevent catastrophic forgetting? **A:** Use rehearsal (replay old task examples), regularization (EWC), or architecture isolation (task-specific adapters).
- **Q:** What's the advantage of multi-task learning? **A:** Shared representations regularize and improve generalization, especially for data-limited tasks.
- **Q:** How do you detect negative transfer? **A:** Compare test performance fine-tuned vs. training from scratch; if fine-tuned is worse, negative transfer occurred.
- **Q:** Why is learning rate critical in fine-tuning BERT? **A:** High rates corrupt pre-trained knowledge; standard Adam (1e-3) is too high; use 2e-5 to 5e-5.
- **Q:** What does gradient reversal in DANN do? **A:** Multiplies gradients by -lambda to the domain classifier, forcing the encoder to fool the domain classifier.
- **Q:** How does prototypical networks compute distance to new class? **A:** Computes mean (prototype) of K support examples, measures query distance to prototype, predicts nearest class.
