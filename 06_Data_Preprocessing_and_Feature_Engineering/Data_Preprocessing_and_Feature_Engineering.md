# Data Preprocessing and Feature Engineering

📺 **Video Lecture:** https://youtu.be/cDMa4ukiDF0


## Interview Anchor
- **Missing Data Mechanisms:** MCAR (completely random), MAR (random given observed), MNAR (depends on unobserved values)
- **Feature Scaling:** Transforming features to comparable ranges; essential for distance-based and gradient-descent algorithms
- **Feature Leakage:** Information from outside training data or future time leaking into model; breaks generalization

## Key Concepts Overview
Preprocessing and feature engineering are often underestimated, but they frequently determine model success more than algorithm choice. Raw data rarely works well directly—values may span different scales, categories need encoding, missing values require principled handling, and relationships between features can be made explicit through engineering. The quality of features directly impacts model performance: good features can be learned by simple models, while poor features stymie sophisticated ones. This is why domain experts often outperform algorithm specialists: they understand what features matter and how to construct them.

Interviewers test this knowledge because it separates practitioners who understand end-to-end ML pipelines from those who focus only on model training. Data leakage, a subtle but catastrophic mistake, reveals whether you think carefully about causality and temporal structure. Handling imbalanced data and understanding bias-variance implications of different preprocessing choices shows maturity in practical ML.

### Q1: Explain missing data mechanisms (MCAR, MAR, MNAR) and their implications.

**A:** Missing Completely At Random (MCAR): probability a value is missing doesn't depend on any observed or unobserved variables. Example: data entry error affects 5% of all values randomly. Implication: listwise deletion (removing rows with any missing values) is unbiased, just loses power. Multiply imputation works well. Missing At Random (MAR): missingness depends on observed variables but not the missing value itself. Example: older people (observed age) are less likely to report income (missing). Implication: simple deletion biases estimates (younger people's incomes overrepresent), but imputation using observed features works. Use multiple imputation or maximum likelihood estimation that accounts for MAR. Missing Not At Random (MNAR): missingness depends on the missing value itself. Example: people with high income avoid reporting it. Implication: deletion and standard imputation are biased—bias depends on relationship between missingness and true value. Requires modeling missingness mechanism explicitly (sensitivity analysis, domain knowledge). In practice: (1) assess mechanism via missing data patterns (do features with missing values cluster?), (2) test MCAR assumption (e.g., compare observed values for those with missing in other columns), (3) use appropriate method (deletion if MCAR and small fraction, imputation if MAR, sensitivity analysis if MNAR). Interviewers appreciate recognizing that blindly deleting missing values can introduce bias; understanding missingness mechanisms shows thoughtful analysis.

### Q2: Compare imputation strategies for missing data.

**A:** Simple deletion: remove rows with any missing values (listwise) or remove columns with >X% missing. Pros: simple, unbiased under MCAR. Cons: loses data, biased under MAR, cannot use if many columns have some missing. Mean/median imputation: replace with column average (mean for continuous, mode for categorical). Pros: simple, preserves sample size, works OK for MCAR. Cons: underestimates variance (imputed values are deterministic, similar values cluster), biases correlations, doesn't use relationships with other variables. Forward/backward fill: for time series, carry forward last observed value (forward fill) or use next value (backward fill). Pros: respects temporal structure. Cons: assumes values stay constant (often wrong), can't use for beginning/end. k-nearest neighbors (KNN) imputation: for each missing value, find k nearest neighbors (using complete variables), average their values. Pros: uses relationships with other variables, reasonable estimates. Cons: computationally expensive, sensitive to distance metric, doesn't generate new variability. Multiple imputation: generate M plausible completed datasets using model (e.g., multivariate normal, regression, chained equations), fit model to each, combine estimates via Rubin's rules. Pros: properly accounts for uncertainty (standard errors include imputation uncertainty), unbiased under MAR. Cons: complex, computationally intensive. Deep learning imputation: neural networks learn feature relationships to predict missing values. Pros: flexible, captures complex relationships. Cons: requires substantial data, interpretability harder, overfitting risk if many values missing. Practical recommendation: if MCAR and <5% missing, simple deletion or mean imputation OK; if MAR and substantial missing, use multiple imputation; if MNAR, combine deletion/imputation with sensitivity analysis. Interviewers value understanding that imputation method affects downstream analysis; reporting imputation approach is essential transparency.

### Q3: Explain outlier detection and treatment strategies.

**A:** Outliers are extreme values that may be errors, rare events, or genuine phenomenon. Detection: (1) statistical: Z-score > 3 (assuming normal distribution), (2) IQR method: values beyond Q1 - 1.5×IQR or Q3 + 1.5×IQR (robust, doesn't assume distribution), (3) isolation forest: assigns outlier score based on tree isolation difficulty, (4) local outlier factor (LOF): anomaly based on local density deviation, (5) Mahalanobis distance: accounts for correlation structure. Treatment: (1) deletion: remove outliers (use if clearly erroneous). Cons: loses information, biases estimates if true extremes. (2) Capping: replace with extreme value from distribution (e.g., 99th percentile). Pros: preserves sample size. Cons: creates artificial clustering at cap value. (3) Transformation: apply log, sqrt, Box-Cox to reduce skewness, making outliers less extreme. Pros: principled, maintains rank. Cons: interpretation changes. (4) Separate model: model outliers separately (mixture model, robust regression). Pros: doesn't discard information, captures true extreme phenomenon. Cons: complex. (5) Keep as is: outliers may be real; model robustness (robust regression, tree-based models) handles them. Cons: may hurt some algorithms (distance-based, neural networks sensitive to scale). Context matters: in anomaly detection, outliers are the target; in regression, they may be noise. Always visualize (box plots, scatter plots) before deciding. Robust methods (median, MAD, Huber regression) are automatically less sensitive to outliers without explicit treatment. Interviewers appreciate thoughtful analysis: outliers shouldn't be reflexively deleted; understanding context and using appropriate methods shows maturity.

### Q4: Explain feature scaling and normalization methods.

**A:** Feature scaling transforms features to compatible ranges; essential because many algorithms (distance-based KNN, gradient descent, neural networks) are sensitive to scale. Standardization (Z-score normalization): (x - mean) / std; results in mean = 0, std = 1. Pros: centers data, makes interpretation easier (how many std from mean), compatible with assumption of normal distribution. Cons: sensitive to outliers (outliers affect mean/std). Min-max normalization: (x - min) / (max - min); results in range [0, 1]. Pros: interpretable (fraction of range), bounded, preserves relationships. Cons: very sensitive to outliers (min/max change drastically), scaling depends on data range (new unseen values may fall outside [0,1]). Robust scaling: (x - median) / IQR; insensitive to outliers. Pros: robust. Cons: less standard, may be less stable with small data. Mean absolute deviation (MAD) scaling: (x - median) / MAD. Unit norm scaling: x / ||x||; makes each sample have unit Euclidean norm. Pros: useful for sparse data. Log scaling: log(x); reduces skewness. Pros: interpretable for multiplicative relationships. Cons: undefined for x ≤ 0. When to scale: (1) algorithms with distance metrics (KNN, K-means, SVM) should always be scaled, (2) gradient-descent algorithms benefit from scaling (condition number improves), (3) regularized methods (Ridge, Lasso, elastic net) are affected by scale (feature with larger scale gets smaller coefficient), (4) tree-based models don't require scaling (splits work on relative values). When not: tree-based models, coefficient interpretation in unscaled space. Practical approach: standardization is default (works well in most cases), min-max for bounded features (probabilities, pixel values), robust scaling if many outliers. Always fit scaler on training data, apply to test/validation data to prevent leakage.

### Q5: Explain categorical encoding methods and their tradeoffs.

**A:** One-hot encoding: create binary column for each category. Example: color=[Red, Blue, Green] becomes [1,0,0], [0,1,0], [0,0,1]. Pros: interpretable, tree-based models handle it well, no ordinal assumption. Cons: increases dimensionality (k categories → k columns, k-1 if dropping one to avoid collinearity), sparse data, can't use with high-cardinality features (hundreds of categories). Label encoding: map categories to integers 0,1,2,... Pros: low dimensionality, interpretable. Cons: implies order (model thinks Blue > Red), inappropriate for unordered categories, tree-based models may split incorrectly (order sensitive). Target encoding (mean encoding): replace category with mean target value. Example: if Red → avg(y | color=Red). Pros: supervised (uses target information), reduces dimensionality, high-cardinality features become small features. Cons: target leakage risk (must use cross-validation or out-of-fold encoding), prone to overfitting (rare categories have noisy estimates), may not generalize to new categories. Ordinal encoding: assign meaningful integers to ordered categories (e.g., Low=1, Medium=2, High=3). Pros: captures order. Cons: choice of spacing matters. Hashing/binary encoding: convert to bits. Pros: handles new categories naturally. Cons: interpretability loss. Frequency encoding: map to frequency of category. Pros: simple. Cons: loses information, multiple categories get same encoding. Practical guide: (1) unordered categories → one-hot (if low cardinality) or target encoding (if high cardinality), (2) ordered categories → ordinal encoding, (3) tree-based models → any method (trees don't care), (4) neural networks → one-hot, (5) high-cardinality features → target encoding (careful with leakage). Avoid target leakage by applying encoding only on training data, using cross-validated estimates for test data. Interviewers value understanding that encoding choice affects model behavior; thoughtful selection based on data characteristics and algorithm shows expertise.

### Q6: Explain feature selection and its variants: filter, wrapper, embedded methods.

**A:** Feature selection removes irrelevant features, reducing dimensionality, improving interpretability, reducing overfitting, and speeding up training. Filter methods evaluate features independently without model. Univariate statistics: (1) correlation with target (linear relationships), (2) mutual information (nonlinear relationships), (3) chi-square test for categorical targets, (4) ANOVA F-statistic for continuous targets. Pros: fast (O(features)), independent of downstream model. Cons: ignores feature interactions, no selection of feature subsets. Wrapper methods train model repeatedly with different feature subsets. Forward selection: start with no features, iteratively add feature that improves model most. Backward elimination: start with all features, iteratively remove feature that hurts least. Recursive feature elimination (RFE): train model, remove feature with smallest coefficient (or importance), repeat. Pros: accounts for feature interactions, selects subsets. Cons: expensive (many model trainings), prone to overfitting (selecting based on test performance), can be unstable. Embedded methods incorporate feature selection into training. Regularization: L1 (Lasso) drives coefficients to zero (implicit feature selection), L2 (Ridge) shrinks but doesn't zero out. Tree-based feature importance: features splitting high in trees get high importance (information gain, Gini importance). Pros: efficient (one training), stable, model-specific. Cons: tree importance can be biased toward high-cardinality features. Practical approach: (1) start with filter methods for quick overview, (2) use regularization (Lasso) for linear models, (3) use tree importance for tree models, (4) use wrapper methods (RFE) if budget allows and overfitting is concern. Dimensionality reduction (PCA) is alternative: instead of selecting, project to lower-dimensional space. Key principle: avoid using test set for feature selection (causes leakage); use cross-validation when wrapping. Interviewers value understanding that feature selection affects interpretability and generalization; explaining reasoning for keeping/removing features shows thoughtfulness.

### Q7: Explain feature extraction and creation of new features.

**A:** Feature extraction creates new features from existing ones, often nonlinear transformations revealing hidden structure. Polynomials: add x², x³, xy, etc. Pros: enable linear models to fit nonlinear relationships. Cons: explosion in dimensionality, risk of overfitting. Interactions: multiply features (xy captures effect of both). Pros: model how features affect each other. Cons: combinatorial explosion, hard to interpret. Binning/discretization: convert continuous to categorical (e.g., age → age groups). Pros: captures nonlinear relationships, easier interpretation. Cons: information loss, arbitrary boundaries. Basis expansion: transform using basis functions (sin, cos, RBF). Pros: flexible, enables kernel methods. Cons: unintuitive. Log transformation: log(x) useful for skewed data, multiplicative relationships, monetary values. Domain-specific features: combine domain knowledge with data. Example: in NLP, extract count of adjectives from text; in computer vision, compute SIFT descriptors. Pros: leverage expertise, interpretable. Cons: labor-intensive, may not generalize. Embedding extraction: use pre-trained models (word2vec, ImageNet embeddings) as features. Pros: capture semantic meaning. Cons: requires external model, less interpretable. Automated feature engineering: tools (Featuretools, TSFRESH) automatically generate candidates, then select. Pros: less manual work. Cons: may miss domain-specific features. Practical approach: (1) start simple (raw features + linear model), (2) add interactions/polynomials if underfit, (3) use domain knowledge to craft interpretable features, (4) use automation for high-dimensional settings. Interviewers value understanding that good features matter more than model choice; thoughtful feature creation demonstrates deep domain understanding. Key insight: simple models with good features outperform complex models with poor features.

### Q8: Explain data leakage and provide examples.

**A:** Data leakage: information from outside training data or from future (relative to prediction time) leaks into training, causing artificially inflated performance estimates that don't generalize. Types: (1) test set contamination: fitting preprocessing (scaling, imputation, feature selection) on full dataset including test. Fix: fit only on training data, apply to test. (2) Target leakage: target information reaches feature space. Example: predicting credit default, feature is "customer called debt collector"—this happens after default, not before. Fix: ensure all features temporally precede target. (3) External data leakage: information available only at training time, not deployment. Example: predicting house price using future construction—not available at prediction time. Fix: validate that features are available at prediction time. (4) Temporal leakage: time-series models trained on future data. Example: using all data to scale features for time-series prediction. Fix: use anchored forward-chaining (scale each fold using only prior data). (5) Information from other targets: predicting spam, features include "user reported this email as spam" (circular). Fix: use only input features, not auxiliary targets. (6) Group leakage: samples from same group in train and test. Example: predicting movie success, multiple samples from same movie in both sets. Fix: group train/test splits by group. Detection: (1) model overfits despite regularization, (2) performance on test much worse than validation, (3) features in production don't match those in training, (4) careful temporal analysis in time-series, (5) domain review: if feature seems too good, it probably is. Prevention: (1) separate train/test before any preprocessing, (2) think causally: can feature be known at prediction time?, (3) use pipelines (sklearn Pipeline, feature_engine) that apply transformations within cross-validation folds. Interviewers care deeply about leakage awareness; mentioning leakage in project discussion is critical credibility marker.

### Q9: Explain handling imbalanced data: oversampling, undersampling, SMOTE.

**A:** Imbalanced data: target classes have skewed distribution (e.g., 1% positive, 99% negative in fraud detection). Naive training leads to models that predict majority class always (high accuracy, useless). Undersampling: remove majority class samples to balance. Pros: simple, fast training (smaller dataset). Cons: loses information, risky if already small dataset, may remove important patterns. Random undersampling: randomly removes. Stratified undersampling: removes while maintaining feature distributions. Oversampling: duplicate minority class to balance. Pros: simple, keeps all data. Cons: creates identical samples (overfitting risk), inflates dataset size. Random oversampling: duplicate randomly. SMOTE (Synthetic Minority Over-sampling Technique): generate synthetic minority samples. For each minority sample, find k nearest neighbors (minority class), interpolate between them: x_synthetic = x_minority + random × (x_neighbor - x_minority). Pros: generates diverse samples (not just duplicates), better than simple oversampling. Cons: can create unrealistic synthetic samples, sensitive to parameter k, still may overfit in small-data regimes. Variants: borderline SMOTE (focus on samples near decision boundary), ADASYN (weights SMOTE by difficulty). Cost-sensitive learning: assign higher misclassification cost to minority class. Pros: doesn't change data, natural for many models. Cons: requires tuning cost ratio. Class weights: set class_weight parameter in models (scikit-learn, many others support this). Pros: simple, avoids data manipulation. Cons: may require tuning. Threshold tuning: change decision threshold (predict positive if prob > threshold, default 0.5). Pros: simple, no data change. Cons: doesn't change model internals, trades precision/recall. Ensemble methods: combine multiple imbalanced models (bagging, boosting with minority focus). Practical approach: (1) start with class weights or cost-sensitive learning, (2) if not enough, try SMOTE, (3) be careful about overfitting (validate properly), (4) use appropriate metrics (F1, AUC, precision-recall curves, not accuracy), (5) test on real imbalanced distribution (don't balance test set). Interviewers value understanding that accuracy is misleading for imbalanced data; knowing multiple approaches and their tradeoffs shows expertise.

### Q10: Explain train/test/validation split strategies and cross-validation.

**A:** Train/test split: divide data (typically 70/30 or 80/20) randomly, train on train set, evaluate on test set. Pros: simple, fast. Cons: single split variance-dependent, may not represent full data variability. Stratified split: ensure class distribution (for classification) or target distribution (for regression) matches in train/test. Pros: balanced representation. Temporal/group split: for time-series, split chronologically (train on past, test on future); for grouped data, split by group (no leakage between groups). k-fold cross-validation: split data into k folds, train on k-1 folds, evaluate on left-out fold, repeat k times, average metrics. Pros: uses all data for both training and evaluation, estimates generalization well, not luck-dependent. Cons: k times computational cost. Practical k: 5 or 10 folds (lower k = more bias and less variance in estimate; higher k = less bias, more variance, more computation). Stratified k-fold: maintains class distribution in each fold. Time-series cross-validation: expanding window (train on 1..t, test on t+1), rolling window (train on t-w..t, test on t+1). Nested cross-validation: outer loop for generalization estimate, inner loop for hyperparameter tuning (avoids biased estimates). Leave-one-out (LOO): k=n (each fold has one sample). Pros: unbiased. Cons: very expensive, high variance. Practical approach: (1) use stratified k-fold for classification (default), (2) use time-series split for time-series, (3) use stratified train/test split + cross-validation for quick iteration, (4) avoid using test set during model selection (use validation set or cross-validation on training set). Critical: fit preprocessing (scaling, imputation, feature selection) only on training fold/data, evaluate on validation/test independently. Interviewers value understanding that cross-validation accounts for variance in performance estimates; explaining why multiple splits matter shows grasp of statistical principles.

### Q11: Explain multicollinearity and its detection (VIF).

**A:** Multicollinearity: features are linearly dependent or highly correlated. Consequences: (1) unstable regression coefficients (small data change causes large coefficient changes), (2) wide confidence intervals (high standard errors), (3) difficulty interpreting feature importance (which feature matters?), (4) numerical instability (near-singular matrix), (5) reduced predictive power (correlated features provide redundant information). Detection: (1) correlation matrix: compute pairwise correlations, look for |r| > 0.8 or 0.9 as threshold. Pros: simple, interpretable. Cons: only detects pairwise, misses higher-order dependencies. (2) Variance Inflation Factor (VIF): VIF_j = 1 / (1 - R²_j), where R²_j is R² from regressing feature j on all others. High VIF (>5 or 10) means feature j is well-predicted by others (highly collinear). Pros: accounts for multivariate relationships. Cons: expensive to compute for high dimensions, threshold somewhat arbitrary. (3) Eigenvalue analysis: compute eigenvalues of X^T X (design matrix correlations). Many small eigenvalues indicate multicollinearity (low-rank structure). Handling: (1) remove one of correlated features (simplest, may lose information). (2) Combine features (e.g., average, PCA). Pros: preserves information. Cons: interpretation harder. (3) Regularization: L2 (Ridge) distributes weight among correlated features; L1 (Lasso) selects one. Pros: maintains all features, improves stability. Cons: interpretation changes. (4) Domain knowledge: if features are indeed measuring similar thing, combination may be appropriate. Practical: (1) compute correlation matrix early in EDA, (2) if high correlation, investigate (is it expected?), (3) use regularization as default (Ridge or Elastic Net), (4) remove highly redundant features if interpretability is priority. In tree-based models, multicollinearity doesn't hurt predictions (trees split on any correlated feature), but may affect feature importance estimates. Interviewers appreciate understanding consequences; mentioning VIF and regularization shows statistical sophistication.

### Q12: Explain polynomial features and interaction terms.

**A:** Polynomial features: add x², x³, ... for each feature. Effect: enable linear models to fit nonlinear relationships. Example: degree-2 polynomial on 3 features (x₁, x₂, x₃) creates 1 + 3 + 3 = 7 features: [1, x₁, x₂, x₃, x₁², x₂², x₃²] plus interactions. Pros: simple to implement, enables flexible models. Cons: dimensionality explosion (degree d on n features creates O(n^d) features), overfitting risk (high-degree polynomials oscillate wildly), harder interpretation. Interaction terms: include products x_i × x_j. Effect: model captures how two features jointly affect target. Example: in e-commerce, price × popularity interaction—effect of price depends on popularity (expensive items sell well if popular). Pros: can greatly improve predictions, often interpretable (domain experts expect interactions). Cons: adds complexity, explosion in dimension with many features (n choose 2 interactions = O(n²)). Automated interaction detection: domain expertise or statistical tests (e.g., significant interaction p-value in ANOVA). Practical approach: (1) start without polynomials/interactions, (2) add if underfitting, (3) be selective (add only reasonable interactions, not all combinations), (4) use regularization (Lasso, Ridge) to avoid overfitting when adding many features, (5) use domain knowledge to identify meaningful interactions. Interviewers appreciate selective feature engineering—blindly adding all polynomials/interactions shows poor judgment; thoughtful selection based on domain understanding demonstrates expertise. In tree-based models, polynomials aren't needed (trees learn them implicitly); in linear models, they're essential for nonlinearity.

### Q13: Explain handling skewness and kurtosis in features.

**A:** Skewness: asymmetry of distribution. Right-skewed (positive): long tail on right (mean > median). Left-skewed (negative): long tail on left (mean < median). Symmetric: skewness ≈ 0. Consequences: (1) outlier sensitivity (skewed data has extreme values), (2) assumption violations (many methods assume normality), (3) scale-dependent (skewed features may dominate standardization). Kurtosis: tail heaviness. High kurtosis: heavy tails (many extreme values), low kurtosis: light tails (fewer extremes). Consequences: (1) outlier sensitivity, (2) variance estimation issues. Treatments: (1) Log transformation: log(x) reduces right skewness. Pros: interpretable (multiplicative becomes additive). Cons: only for x > 0, less effective for heavy left skew. (2) Box-Cox transformation: λ tuned to minimize skewness. Pros: flexible, optimal. Cons: less interpretable, λ depends on data. (3) Yeo-Johnson: variant of Box-Cox, handles x ≤ 0. (4) Quantile normalization: map to quantile of normal distribution. Pros: forces normality. Cons: artificial, may hurt relationships. (5) Robust preprocessing: use median/MAD instead of mean/std (less sensitive to extremes). Detection: (1) histograms: visualize distribution, (2) Q-Q plots: compare to normal; deviation indicates non-normality, (3) Shapiro-Wilk test: statistical test for normality, (4) skewness/kurtosis metrics: compute from data. Practical: (1) visualize features (always!), (2) if severely skewed, apply transformation (log-transformation is simple starting point), (3) some algorithms need normality (linear models, neural networks) more than others (trees don't care), (4) decision tree-based models: no transformation needed (trees work on ranks), (5) for continuous features in neural networks: standardization + optionally transformation helps training. Interviewers value recognizing skewness and its consequences; understanding practical fixes demonstrates maturity.

### Q14: Explain the concept of data quality and how to assess it.

**A:** Data quality dimensions: (1) completeness: fraction of non-missing values. Target: 100%, realistic: >95%. (2) Accuracy: values match reality. Assess: domain expert review, comparison with authoritative sources. (3) Consistency: values follow expected format/ranges. Example: age should be positive integers, email should have @. (4) Validity: values in valid domain (e.g., blood type in {A, B, AB, O}). (5) Uniqueness: no unintended duplicates. (6) Timeliness: data is recent, current. Assessment tools: (1) data profiling: compute summary statistics (min, max, mode, missing count), (2) null analysis: identify missing value patterns, (3) outlier analysis: detect extreme values, (4) distribution plots: histograms, box plots reveal issues, (5) validation rules: check explicit constraints (age > 0, email format valid), (6) duplicate detection: find identical or near-duplicate rows, (7) consistency checks: relationships between fields (endDate > startDate?). Pandas profiling: automated library generating comprehensive data quality report. Issues and remediation: (1) missing values: appropriate imputation, deletion, (2) duplicates: remove or merge, (3) invalid values: correction, deletion, or reclassification, (4) outliers: investigation, deletion, transformation, (5) inconsistencies: standardization, normalization, (6) encoding issues: correct character encoding. Documentation: record data issues, remediation applied, and assumptions made. Practical: (1) invest time in data quality early (garbage in, garbage out), (2) automate quality checks (unit tests for data), (3) document issues and fixes for reproducibility, (4) involve domain experts (they know what's normal). Interviewers value recognizing data quality as foundational; mentioning systematic assessment and documentation shows professional rigor.

### Q15: Explain pipeline design and avoiding data leakage in practice.

**A:** ML pipeline: sequence of transformations from raw data to predictions. Components: data loading → cleaning/imputation → feature engineering → scaling → feature selection → model training → evaluation. Data leakage risk at each step: (1) preprocessing fitted on full data (train + test), (2) feature engineering uses future information, (3) feature selection on full dataset, (4) hyperparameter tuning using test set. Best practice: build pipeline that fits preprocessing only on training data, applies to test/validation data. Scikit-learn Pipeline: encapsulates preprocessing + model. Example: Pipeline([('scaler', StandardScaler()), ('model', LogisticRegression())]). Fit on training data, transform + predict on test data—scaler fits only on training data, preventing leakage. Benefits: (1) ensures transforms applied consistently, (2) prevents leakage (transforms fit only on training), (3) simplifies code, (4) reusable, (5) serializable (save/load entire pipeline). Cross-validation: use cross-validation with pipelines to estimate generalization. sklearn's cross_validate applies pipeline correctly: fits within each fold, prevents leakage. Custom pipelines: for complex workflows (multiple feature groups with different preprocessing), build custom pipeline or use ColumnTransformer. Monitoring in production: pipeline applies same transforms to new data. Important: save preprocessing artifacts (scaler, feature selectors) during training, apply exactly same artifacts during serving. Common pitfalls: (1) fitting scaler on all data before splitting, (2) using test set to select features, (3) hyperparameter tuning on test set (use validation set or cross-validation), (4) applying different scaling to different datasets, (5) forgetting to apply same preprocessing in production. Validation strategy: (1) cross-validation estimates unbiased generalization, (2) held-out test set confirms cross-validation estimate, (3) production performance validates real-world effectiveness. Interviewers highly value pipeline awareness; explaining how you prevent leakage and ensure reproducibility differentiates strong candidates.

## Interview Cheatsheet

**Key Terms:**
- **MCAR:** Missing Completely At Random; deletion unbiased
- **MAR:** Missing At Random; depends on observed variables; imputation with observed variables works
- **MNAR:** Missing Not At Random; depends on missing value; requires sensitivity analysis
- **Standardization:** (x - mean) / std; results in mean=0, std=1
- **Min-max Normalization:** (x - min) / (max - min); results in range [0, 1]
- **One-hot Encoding:** Convert categorical to binary columns; one column per category
- **Label Encoding:** Map categories to integers; can incorrectly imply order
- **Target Encoding:** Replace category with mean target value; prevents leakage via cross-validation
- **Feature Selection:** Remove irrelevant features; filter (univariate), wrapper (model-based), embedded (regularization)
- **Feature Engineering:** Create new features; polynomials, interactions, transformations
- **Data Leakage:** Information from outside or future leaks into training; breaks generalization
- **SMOTE:** Synthetic Minority Over-sampling Technique; generates realistic synthetic minority samples
- **Class Weights:** Assign higher cost to minority class in loss function; prevents training instability
- **k-fold Cross-validation:** Split data into k folds; average metrics across folds; reduces variance of estimate
- **Multicollinearity:** Features highly correlated; causes unstable coefficients, wide confidence intervals
- **VIF:** Variance Inflation Factor; VIF > 5-10 indicates multicollinearity
- **Stratified Split:** Maintain class distribution in train/test; prevents imbalance
- **Data Leakage Prevention:** Fit preprocessing only on training data, apply to test; use pipelines

**Rapid-Fire Q&A:**
- **Q: Why fit scaler only on training data?** **A:** Prevents leakage; test set scaling parameters must not depend on test data itself
- **Q: When should you use target encoding?** **A:** High-cardinality categorical features; always use cross-validation to prevent target leakage
- **Q: What's the difference between oversampling and SMOTE?** **A:** Oversampling duplicates minority samples (overfitting risk); SMOTE generates synthetic samples via interpolation (more diversity)
- **Q: How do you detect multicollinearity?** **A:** Compute VIF (>5-10 indicates high collinearity) or correlation matrix (|r| > 0.8-0.9)
- **Q: Why is stratified train/test split important?** **A:** Maintains class distribution in both sets; prevents imbalance where one set is skewed toward majority class
