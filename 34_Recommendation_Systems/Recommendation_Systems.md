# Recommendation Systems

📺 **Video Lecture:** https://youtu.be/8-Y8X7ZOD2s


## Interview Anchor
- **Collaborative Filtering:** Recommending items based on user similarity: if users A and B rated similar items similarly, recommend A's liked items to B
- **Content-Based Filtering:** Recommending items similar to those the user previously liked, based on item features
- **Hybrid Systems:** Combining collaborative and content-based approaches to leverage strengths of both

## Key Concepts Overview
Recommendation systems are among the most commercially valuable ML applications, powering Netflix, Amazon, YouTube, and Spotify. They address a fundamental problem: with millions of items, how do you personalize for each user? Recommendation systems combine collaborative filtering (learn from crowd behavior), content-based filtering (understand item features), and increasingly deep learning (capture complex patterns) to solve this. Understanding recommendation systems is valuable because they involve diverse ML techniques (matrix factorization, neural networks, ranking algorithms, reinforcement learning), face real-world challenges (cold start, exploration-exploitation), and drive significant business value (Netflix credits recommendations with preventing cancellations).

### Q1: What is collaborative filtering and how does user-based vs item-based filtering differ?

**A:** Collaborative filtering recommends items based on user similarity: if users A and B rated movies similarly, items B liked but A hasn't seen are good recommendations for A. Two approaches: **User-based**: (1) find users similar to target user, (2) recommend items those similar users liked. Example: if you and I both rated "Inception" and "The Matrix" highly, I'll recommend to you my other favorite "Interstellar." Similarity computed via cosine similarity of rating vectors. Advantage: intuitive, works well when users have similar tastes. Disadvantage: doesn't scale (comparing to all users is O(n)), relies on user vectors being populated (cold start problem). **Item-based**: (1) find items similar to target user's liked items, (2) recommend those similar items. Example: if you liked "Inception," recommend "Tenet" (similar director, complexity). Similarity computed via item-item correlations (users who liked A also liked B?). Advantage: more scalable (precompute item similarities), better for sparse data. Disadvantage: less intuitive, may recommend similar items repeatedly. In practice: item-based is more scalable and is standard in production systems. Both are memory-based (no explicit training), computing similarities on-demand from rating matrix, making them fast but requiring full ratings matrix in memory.

### Q2: Explain matrix factorization (SVD, ALS) and why it works for recommendations.

**A:** Matrix factorization decomposes the user-item rating matrix (m users × n items) into two lower-rank matrices: U (m users × k latent factors) and V (n items × k latent factors). A rating is approximated as: rating(u,i) ≈ U[u] · V[i]. The factorization finds latent factors representing user preferences and item characteristics. Example: factors might represent movie genres (action, romance, sci-fi); U[user] indicates how much user likes each genre, V[movie] indicates how much movie features each genre. **SVD (Singular Value Decomposition)** solves this via eigenvalue decomposition, finding optimal factorization. But SVD requires dense matrices; recommendation matrices are sparse (users rate few items). **ALS (Alternating Least Squares)** is iterative: (1) initialize V randomly, (2) fix V, solve for U (least squares), (3) fix U, solve for V, (4) repeat until convergence. ALS handles sparsity elegantly and parallelizes well. Benefits of factorization: (1) **dimensionality reduction** - compress massive matrix to small factors, (2) **generalization** - learn patterns from observed ratings, predict unobserved ones, (3) **efficiency** - O(k) space per user/item instead of O(n). ALS is computationally efficient and used in production systems (Spark MLlib, Databricks). Limitation: doesn't use item/user features; purely collaborative.

### Q3: What is content-based filtering and when is it preferable to collaborative?

**A:** Content-based filtering recommends items similar to those the user liked, using item features. Process: (1) represent each item as feature vector (genres, actors, budget, release year for movies), (2) represent user preferences as weighted feature vector (learned from user's rated items), (3) score unrated items by similarity to user preference vector, (4) recommend top-scoring items. Example: user likes "Inception" (features: sci-fi, heist, Christopher Nolan); recommend "Tenet" (similar features). Content-based advantages: (1) **cold start for items** - new items can be recommended immediately if features are known, (2) **transparency** - explanations are feature-based ("recommended because it's sci-fi"), (3) **serendipity control** - can adjust to recommend similar or diverse items. Disadvantages: (1) **feature engineering** - requires manual feature definition or extraction, (2) **limited novelty** - tends to recommend items very similar to past likes, (3) **cold start for users** - new users with no history are hard to recommend for. Collaborative vs content-based: (1) **collaborative is better when** - users have rated many items (matrix is populated), item features are complex/implicit, (2) **content-based is better when** - item features are rich and explicit, new items/users are frequent, transparent recommendations are needed. Hybrid approaches combine both: use collaborative for popular items (rich rating data), content-based for new items (limited ratings).

### Q4: What is the cold start problem and how do you solve it?

**A:** Cold start is the challenge of making recommendations when data is limited: (1) **new users** - no rating history, collaborative filtering can't find similar users, (2) **new items** - few ratings, can't assess quality or find similar items, (3) **new system** - initially no ratings at all. Cold start is serious: new users arrive daily; if recommendations suck, they churn. Solutions: (1) **content-based** - use item features to recommend similar items, works for new items immediately, (2) **demographic features** - recommend based on user demographics (age, location) before personalization kicks in, (3) **active learning** - ask new users explicitly for preferences ("like these movies?"), (4) **hybrid** - combine collaborative and content-based, (5) **exploration** - recommend diverse items to gather ratings, (6) **knowledge-based** - ask questions to learn user preferences ("budget: high, genre: action?"). For new items, common approach: show based on popularity/recency initially, collect ratings, then personalize. For new users: ask for preferences explicitly (rating 5 items), then leverage those to seed recommendations. Context matters: Netflix can ask for movie preferences explicitly (users expect this), a product recommendation system can't (user patience is limited). Best practice: acknowledge cold start explicitly, use multiple data sources (collaborative, content, demographics, context), and monitor new user/item performance separately.

### Q5: What is implicit vs explicit feedback and how does it change recommendations?

**A:** **Explicit feedback** is direct ratings/reviews (5-star rating, thumbs up/down). Users consciously express preferences; data is high-quality but sparse (users rate few items). **Implicit feedback** is inferred from behavior: clicks, purchases, watch time, dwell time, skips. Implicit feedback is sparse (billions of users × items) but dense in observed user actions. Example: "User watched 90% of movie = likely = strong positive. User skipped after 2 mins = dislike. User added to wishlist = strong interest." Implicit vs explicit differences: (1) **feedback scale** - explicit is ordered (1-5 stars); implicit is implicit confidence (watching 100% is stronger than 50%), (2) **negative feedback** - explicit includes negative ratings; implicit lacks true negatives (non-interactions could mean disinterest or unawareness), (3) **volume** - implicit is much denser (clicks >> ratings), (4) **biases** - explicit suffers selection bias (users who rate are different from general population); implicit reflects behavior of all users. Algorithms differ: explicit uses standard collaborative filtering, implicit typically uses weighted least squares (negative interactions weighted lower) or BPR (Bayesian Personalized Ranking, which ranks observed items above unobserved). In practice: most systems use implicit (more data available). YouTube uses watch time, Netflix uses viewing history. Implicit is harder to handle (no true negatives) but more practical at scale. State-of-the-art combines both: explicit ratings for engaged users, implicit signals for all users.

### Q6: Explain evaluation metrics for recommendations (precision@K, recall@K, NDCG, MAP, MRR).

**A:** **Precision@K** - fraction of top-K recommendations that user interacted with. E.g., if you recommend 10 movies and user watches 2, precision@10 = 0.2. Measures: "how many recommendations are good?" High precision means few false positives. **Recall@K** - fraction of user's interactions that appear in top-K recommendations. E.g., if user watched 100 movies and 8 appear in recommended 10, recall@10 = 0.08. Measures: "did you recommend movies user cares about?" High recall means few false negatives. Precision vs recall: precision asks "are recommendations good?" recall asks "did you cover user's interests?" High precision, low recall: recommend only movies you're sure user likes (many movies missed). High recall, low precision: recommend many movies (some are irrelevant). **NDCG (Normalized Discounted Cumulative Gain)** accounts for ranking: top recommendations matter more than lower. DCG = Σ (rating / log(position)). Example: if user likes position 1 (weight 1), then position 3 (weight 0.63), then position 10 (weight 0.42), DCG = 1 + 0.63 + 0.42. NDCG normalizes by ideal DCG (user liked top K items). Accounts for ranking. **MAP (Mean Average Precision)** - average of precision@K computed at each K where user interacted. If user interacted at positions 2, 5, 8: AP = (P@2 + P@5 + P@8) / 3. **MRR (Mean Reciprocal Rank)** - 1 / position of first correct recommendation. If correct recommendation is at position 3, MRR = 1/3. Common: precision/recall for binary feedback (clicked or not), NDCG for ranked feedback (ratings 1-5), MRR for ranking where order matters (search results). Choose metric based on application goal: precision for no false positives, recall for coverage, NDCG/MAP for ranking quality.

### Q7: What are deep learning approaches for recommendations (NCF, Wide & Deep, DeepFM)?

**A:** Deep learning models capture complex non-linear interactions in recommendation. **NCF (Neural Collaborative Filtering)** replaces matrix factorization with neural network: (1) embed user and item to learned vectors, (2) concatenate embeddings, (3) pass through MLP (multi-layer perceptron), (4) output prediction score. Network learns user-item interactions implicitly through hidden layers. Benefits: flexible, can incorporate item/user features. **Wide & Deep** combines memorization and generalization: (1) **wide component** - linear model with feature interactions (captures memorized patterns, e.g., "user X always likes action movies"), (2) **deep component** - neural network (learns generalizable patterns, e.g., "similar users like similar items"). Both feed into final prediction. Intuition: wide captures "this user always likes X," deep learns "users like you tend to like Y." Used in Google Play recommendations. **DeepFM** combines factorization machines and deep learning: (1) **FM component** - second-order feature interactions (user_id * item_id, user_id * genre), (2) **deep component** - high-order interactions through MLP. Both learn from same embeddings, combined output. FM captures pairwise interactions, deep learns higher-order. All three outperform matrix factorization because: (1) **non-linearity** - neural networks capture complex patterns, (2) **feature interactions** - explicitly model user-item interactions, (3) **side information** - incorporate auxiliary features (user age, item genre). Tradeoffs: more complex (harder to train, tune, deploy), require more data, less interpretable. Use deep learning when data/resources available and factorization performance plateaus.

### Q8: What are session-based recommendations and how do they differ from user-based?

**A:** Session-based recommendations predict the next item the user will interact with in the current session, without requiring user history. Session: a sequence of user actions (clicks, purchases) within a time window. Example: user enters store, looks at shoes, then jackets, then hats. Next recommendation should consider current session context. Session-based differs from user-based: (1) **user-based** - learns long-term preferences (who is this user?), (2) **session-based** - learns short-term intent (what is the user doing right now?). Algorithms: (1) **GRU4Rec** - RNN with GRU cells treating sessions as sequences, (1) **Transformer-based** (SASRec) - attention mechanism captures dependencies within session. Example: user clicked {shoes, jacket, hat}. GRU4Rec learns "users clicking shoes typically click jacket next," and outputs jacket. Benefits: (1) **no user history required** - works for anonymous users, (2) **context awareness** - captures current intent, (3) **real-time** - no need to store user profiles. Limitations: (1) **no personalization** - doesn't learn individual preferences over time, (2) **cold start remains** - new sessions have no history, (3) **limited scope** - only current session available. Combined approach: (1) session-based for immediate next item (real-time), (2) user-based for longer-term preferences (recommendations tab). Session-based is standard for e-commerce (Amazon, eBay) where user intent changes within a shopping session, and for content discovery (Netflix) where watch session indicates interest.

### Q9: What is the exploration-exploitation trade-off and how do multi-armed bandits solve it?

**A:** Exploitation: recommend items you're confident user likes (high predicted rating). Exploration: recommend uncertain items to learn user preferences (reduce uncertainty). Tradeoff: (1) **pure exploitation** - always recommend high-scoring items, but miss discovering what user might enjoy, (2) **pure exploration** - always try new items, but recommend bad items. Real recommendation systems need both. **Multi-armed bandits** (MAB) formalize exploration-exploitation. Each item is an "arm," pulling an arm = recommending, reward = user interaction. Strategies: (1) **epsilon-greedy** - exploit with probability 1-epsilon (recommend best item), explore with probability epsilon (recommend random item). Simple, but random exploration is inefficient. (2) **Upper Confidence Bound (UCB)** - confidence depends on exploration. Select arm with highest upper confidence bound (exploit if confident, explore if uncertain). Mathematically: UCB = mean_reward + sqrt(log(t) / count). Arms with fewer pulls have higher confidence bonus, encouraging exploration. (3) **Thompson Sampling** - Bayesian approach. Maintain belief distribution over arm rewards, sample from distribution, select best sampled arm. Over time, true good arms are sampled more often. Benefits: (1) **principled** - balance exploration-exploitation optimally, (2) **regret bounds** - theoretical guarantees on cumulative regret. Use cases: A/B testing (bandit optimization finds winning variant faster), dynamic pricing (explore prices, exploit high-margin prices), news recommendations (explore new stories, exploit popular ones). Tradeoff: exploration has short-term cost (recommend uncertain items) for long-term gain (better personalization). Most production systems use bandits or bandit-inspired exploration (e.g., "explore with 10% probability").

### Q10: Explain the candidate generation + ranking pipeline.

**A:** At scale, scoring all items for every user is expensive (YouTube has billions of videos, millions of users; scoring all combinations is infeasible). Two-stage pipeline: (1) **candidate generation** - retrieve ~1000 promising items from billions, (2) **ranking** - rank 1000 candidates, return top-10. **Candidate generation** uses fast, coarse models: collaborative filtering, content-based, or simple ML. Example: YouTube uses matrix factorization to retrieve videos. Process: compute user embedding, search approximate nearest neighbors in item embedding space, retrieve top-1000 videos. Fast (milliseconds), covers diverse candidates. **Ranking** uses complex models: deep neural networks, incorporating rich features. Process: score 1000 candidates with factors like: personalization (user past behavior), context (time of day, device), diversity (user tired of action movies?), business metrics (engagement, watch time, revenue). Rank by combined score, return top-10. Ranking is expensive per-item but applied to only 1000 items. Pipeline benefits: (1) **efficiency** - candidate generation filters huge space, ranking focuses resources on promising items, (2) **modularity** - swap candidate generator or ranker independently, (3) **quality** - ranking uses rich context impossible in candidate generation stage. Challenges: (1) **cascade effect** - if candidate generation misses good items, ranking can't recover, (2) **coverage** - ensure diverse candidates generated (cold items, niche content), (3) **context incorporation** - ranking uses context (time, user device) but candidate generation doesn't, leading to mismatch. Mitigation: multi-candidate-generation (use multiple generators, merge results), consider context in generation, online learning (retrain frequently with new data).

### Q11: What is feature engineering for recommendations and what are common features?

**A:** Feature engineering creates features from user behavior and item properties for ML models. Common features: **(1) User features**: age, gender, location, account tenure, sign-up device, language, subscription level. **(2) Item features**: genre, duration, release date, popularity, average rating, number of reviews, actor/director names. **(3) Interaction features**: user_watched_genre (is this genre similar to ones user watches?), user_director_similarity, item_popularity_rank. **(4) Contextual features**: time_of_day, day_of_week, device (phone/TV), network type, season. **(5) Historical features**: user_avg_rating (how generous is this user?), user_watch_time (is user patient?), item_completion_rate (do users finish this item?). **(6) Temporal features**: days_since_user_rated_similar_item (recency of similar taste), days_since_item_released, decay (older interactions weighted less). Feature engineering process: (1) **domain knowledge** - what do you know about recommendations that features should capture? (2) **experimentation** - try different features, measure impact, (3) **statistical validation** - ensure features are correlated with outcome and don't leak future information, (4) **embedding** - some features are categorical (genre, actor); use embeddings to represent them, (5) **normalization** - scale features to comparable ranges. Challenges: (1) **feature leakage** - e.g., "user_will_watch" is the target, don't include it as feature, (2) **stale features** - computed offline, may be outdated at prediction time, (3) **coverage** - new users/items lack some features (fill with defaults or learn embeddings), (4) **dimensionality** - thousands of features slow training/serving. Best practice: start simple (user/item IDs), add features incrementally, measure impact, remove low-impact features.

### Q12: How do you handle real-time recommendations with low latency?

**A:** Real-time recommendations must respond in milliseconds (SLA: <100ms typical). Challenges: (1) **fresh data** - incorporate latest user behavior, (2) **scale** - millions of concurrent users, (3) **latency** - complex models too slow. Techniques: (1) **pre-computation** - pre-compute recommendations offline, serve from cache. For each user, compute top-100 candidates nightly, store in fast storage (Redis). At request time, retrieve pre-computed list, apply real-time context (e.g., exclude already-watched), return. Advantage: <10ms latency. Disadvantage: recommendations are 24 hours old. (2) **candidate caching** - cache top candidates per user (computed daily), rank in real-time with latest context. Hybrid of pre-computation and real-time ranking. (3) **approximate nearest neighbors (ANN)** - user/item embeddings indexed in ANN structures (HNSW, Annoy, Faiss). At request time, retrieve user embedding, search ANN index for similar items, return. Milliseconds possible. (4) **lightweight models** - use fast models in serving path (linear models, simple trees), reserve complex models for offline training. (5) **GPU serving** - GPUs process batches of candidate rankings faster than CPUs. (6) **edge caching** - cache recommendations geographically close to users, reduce latency. Implementation: (1) **serving infrastructure** - dedicated microservices for candidate generation and ranking, (2) **monitoring** - track latency, cache hit rate, (3) **A/B testing** - ensure pre-computed/cached recommendations perform as well as online models. Netflix pre-computes nightly, Amazon uses real-time ranking with ANN candidate generation. Tradeoff: pre-computation is faster but stale; real-time is fresher but slower. Hybrid approaches balance both.

### Q13: What is knowledge graph-enhanced recommendations?

**A:** Knowledge graphs represent entities (users, items, attributes) and relationships (user-likes-item, item-has-genre) as graphs. Recommendation uses graph structure to infer preferences. Example: user u liked item i, item i has attribute "sci-fi." Knowledge graph infers: user u might like other sci-fi items. **Graph structure**: nodes are entities (users, movies, actors, genres, directors), edges are relationships. Rich structure enables: (1) **reasoning** - follow paths (u → movie → actor → movie), (2) **semantic similarity** - similar nodes in graph are similar (movies with same director/genre), (3) **auxiliary information** - incorporate metadata that might not be in training data. **Algorithms**: (1) **graph embeddings** - learn embeddings for nodes/edges, similar to word2vec but on graphs (Node2Vec, Graph Autoencoders), (2) **graph neural networks** - propagate information through graph (GAT, GCN), (3) **path-based reasoning** - if user u → movie → actor → movie → recommendation, explain with path. Benefits: (1) **explainability** - recommendations come with explanation ("you watched Avatar directed by James Cameron; we recommend his other film Titanic"), (2) **serendipity** - graph structure finds non-obvious recommendations, (3) **cold start mitigation** - new items with known attributes (actor, director) can be connected in graph. Challenges: (1) **graph construction** - requires curating entity graphs, expensive at scale, (2) **sparsity** - graphs are often sparse (few relations per node), (3) **scalability** - graph algorithms can be computationally expensive. Use cases: movie recommendations (rich graphs: actors, directors, genres), e-commerce (product graphs: brands, categories, reviews), music (artist/genre/tag graphs). State-of-the-art combines: embeddings + knowledge graphs + deep learning.

### Q14: What are transformer-based recommendations (SASRec, BERT4Rec)?

**A:** Transformers (attention mechanisms, self-attention) capture sequential patterns in user behavior. **SASRec (Self-Attentive Sequential Recommendation)**: treat user's item history as sequence, use self-attention to learn which past items influence next item. Process: (1) embed each item in history, (2) apply self-attention (each item attends to all prior items, learning which matter), (3) predict next item. Advantage: flexible attention learns complex temporal patterns. Example: user watched {action, romance, sci-fi}. Self-attention learns: "this user alternates genres" or "genre depends on mood inferred from time of day." **BERT4Rec**: use BERT (bidirectional encoder) on item sequences. Mask random items in sequence, train model to predict masked items (similar to BERT's masked language modeling). Benefits: learns bidirectional context (future items influence interpretation of past), pre-training on raw data (no labels needed). Advantages of transformers: (1) **flexibility** - attention learns complex patterns, (2) **interpretability** - attention weights show which items matter, (3) **state-of-the-art** - transformers achieve top results on recommendation benchmarks, (4) **transfer learning** - pre-train on large data, fine-tune for specific domain. Disadvantages: (1) **computation** - self-attention is O(sequence_length^2), slow for long histories, (2) **requires sequence data** - works best when item order matters (session-based), less for user-user similarities. Practical: use transformers for session-based (clear sequence), combine with collaborative filtering for long-term patterns. Transformers are research-forward; production systems still use simpler models (matrix factorization, gradient boosting) but increasingly adopting transformers.

### Q15: How do you optimize recommendations for different business metrics (engagement, revenue, diversity)?

**A:** Recommendation systems have multiple objectives: (1) **engagement** (clicks, watch time), (2) **revenue** (conversion, AOV), (3) **diversity** (avoid repetition, discovery), (4) **fairness** (all users served equally, all items have chance), (5) **retention** (keep users, prevent churn). These can conflict: promoting high-revenue items might reduce engagement; maximizing diversity might reduce revenue. Techniques: (1) **multi-objective optimization** - define loss with weighted sum: L = w_engagement * L_engagement + w_revenue * L_revenue + w_diversity * L_diversity. Weights control priority. (2) **re-ranking** - generate candidates optimizing one metric, re-rank to optimize others. Example: generate candidates for engagement, re-rank to boost diversity (ensure recommendations are varied). (3) **contextual bandit** - frame as MAB with structured feedback. Context = user/item features, action = recommended item, reward = engagement + revenue + diversity. Bandit learns which actions maximize combined reward. (4) **fairness constraints** - explicitly constrain recommendations to satisfy fairness (e.g., at least 10% of recommendations are from underrepresented categories). (5) **A/B testing** - test different weightings, measure which achieves business goals. Implementation: (1) **model changes** - retrain with new loss weights, (2) **serving changes** - apply re-ranking rules without retraining. Practical example: Netflix wants engagement (watch time) and diversity (avoid recommending only popular titles). Weighting: L = 0.8 * engagement + 0.2 * diversity. Re-ranking: for top-100 candidates, ensure 20% are niche content. A/B test shows: engagement drops 2% but diversity increases 30%; if churn decreases, launch it. Tradeoff: optimizing multiple metrics requires careful tuning; overshooting one metric hurts others (too much diversity reduces engagement).

## Interview Cheatsheet

**Key Terms:**
- **Collaborative Filtering:** Recommendations based on user similarity
- **Matrix Factorization:** Decomposing user-item rating matrix into user/item factors
- **Content-Based Filtering:** Recommendations based on item feature similarity
- **Cold Start:** Challenge of recommending with limited data
- **Implicit Feedback:** Inferred preferences (clicks, watches) vs explicit (ratings)
- **NDCG:** Metric accounting for ranking order in recommendations
- **Candidate Generation + Ranking:** Two-stage pipeline for scalable recommendations
- **Exploration-Exploitation:** Balancing recommending confident items vs learning new preferences
- **Multi-Armed Bandits:** Framework for solving exploration-exploitation trade-off

**Rapid-Fire Q&A:**
- **Q: Should I use collaborative or content-based filtering?** **A:** Both; combine in hybrid. Collaborative learns crowd behavior, content handles new items.
- **Q: How do I handle new users?** **A:** Ask for explicit preferences, use demographic recommendations, leverage session-based context.
- **Q: Why does ALS work better than SVD for recommendations?** **A:** Handles sparsity gracefully, parallelizes well, supports incremental updates.
- **Q: What metric matters most?** **A:** Depends on business. Engagement? Precision. Coverage? Recall. Ranking quality? NDCG.
- **Q: How do I prevent recommendation filter bubbles?** **A:** Multi-objective optimization (add diversity term), re-ranking with diversity constraints, exploration.
