# Ensemble Methods

## Interview Anchor
- **Bagging vs. Boosting:** Bagging trains parallel models on bootstrap samples, reducing variance; boosting trains sequential models on residuals/misclassified samples, reducing bias.
- **Stacking:** Combines diverse models via a meta-learner; more flexible but complex than simple averaging.
- **Model Diversity:** Ensemble strength depends on individual model differences; diverse errors cancel via averaging.

## Key Concepts Overview

Ensemble methods are a cornerstone of modern machine learning, frequently dominating competitions and production systems. This topic tests both theoretical understanding—bias-variance decomposition, why ensembles work—and practical knowledge of specific methods (AdaBoost, gradient boosting, stacking). Interviewers assess whether you understand why ensemble diversity is critical, how different ensemble types address different problems (bagging for variance reduction, boosting for bias reduction), and the implementation details of XGBoost/LightGBM. Additionally, this topic touches on computational trade-offs (ensemble inference is slower than single models) and when simpler baselines suffice. The depth of knowledge here—understanding that dropout is an ensemble method, recognizing over-regularization in boosting—distinguishes strong candidates.

### Q1: Explain the bias-variance decomposition and how ensembles reduce variance via bagging.

**A:** Bias-variance decomposition decomposes expected test error: E[(y - ŷ)²] = Bias(ŷ)² + Var(ŷ) + σ² (irreducible noise). Bias is systematic error (model too simple), variance is sensitivity to training data (model too complex). Bagging reduces variance by training multiple models M_i on bootstrap samples and averaging: ŷ_ensemble = (1/M) ∑ŷ_i(x). If each model has variance σ² and correlations decay with bootstrap variation, ensemble variance is approximately σ²/M—variance decreases with ensemble size (square root improvement). Bias remains unchanged (ensemble of high-bias models still has high bias). Mathematical intuition: each ŷ_i is noisy, but averaging independent noise cancels it. In practice, models aren't fully independent (same algorithm, correlated data), so variance reduction is less than σ²/M, but improvement is substantial. Bagging works best for high-variance, low-bias base models (e.g., deep decision trees, shallow neural networks). Random Forests are bagging applied to trees; each tree has high variance individually, but ensemble averaging reduces variance dramatically. Bagging doesn't address underfitting (high bias)—adding more high-bias models doesn't help. In interviews, explain that bagging/ensembles solve the variance problem; if models underfit, bagging won't help (need boosting or complex models).

### Q2: Explain boosting and how it reduces bias. Describe AdaBoost.

**A:** Boosting trains models sequentially, where each subsequent model focuses on samples the previous models misclassified (reweighting or resampling). This reduces bias: if base models underfit (e.g., shallow trees—weak learners), boosting combines them to fit complex structure. AdaBoost (Adaptive Boosting) is the classical boosting algorithm: (1) initialize equal sample weights w_i = 1/n, (2) train a weak learner (typically decision stump—1-level tree) on weighted data, (3) compute weighted error ε = ∑_{misclassified} w_i, (4) set model weight α = (1/2) log((1-ε)/ε), (5) update weights: w_i ← w_i × exp(-α y_i ŷ_i), increasing weights on misclassified samples. (6) repeat for T rounds. Final prediction: ŷ = sign(∑_t α_t ŷ_t). Intuition: each round focuses on hardest samples; combining weak learners (each > 50% accuracy) yields strong learner. Advantages: (1) theoretically grounded (exponential loss), (2) automatic margin maximization, (3) works with any base learner. Disadvantages: (1) sensitive to noisy data (noise amplified by reweighting), (2) requires careful tuning (learning rate, iterations), (3) slower than bagging (sequential), (4) individual weak learners must be > 50% accuracy. Modern gradient boosting supersedes AdaBoost, but AdaBoost is foundational—understanding it shows theoretical depth. In interviews, mention AdaBoost as historical; gradient boosting is the standard now.

### Q3: Explain the gradient boosting framework and how it generalizes AdaBoost.

**A:** Gradient boosting trains models sequentially to minimize a loss function by fitting each new model to pseudo-residuals (negative loss gradients). Formally, initialize F_0(x) = argmin_c ∑ L(y_i, c) (constant prediction), then iterate: (1) compute pseudo-residuals r_i = -∂L/∂F_{m-1}(x_i), (2) fit base learner h_m to (x_i, r_i), (3) update F_m(x) = F_{m-1}(x) + η h_m(x), where η is learning rate. Intuition: each iteration improves predictions by adding models that capture remaining errors. For squared loss (regression), r_i are actual residuals; for cross-entropy (classification), r_i are loss gradients. Advantages: (1) works with any differentiable loss (L2, cross-entropy, custom), (2) more flexible than AdaBoost, (3) often achieves better generalization, (4) theoretically justified by functional gradient descent. Disadvantages: (1) more hyperparameters (learning rate, number of rounds, loss function), (2) prone to overfitting if not regularized carefully, (3) slower than bagging (sequential). Gradient boosting is the foundation for XGBoost, LightGBM, CatBoost—all modern high-performance libraries. In interviews, explaining gradient boosting as a generalization of AdaBoost and describing the residual-fitting view shows strong understanding. Mention that learning rate trades off step size (smaller ↔ more iterations needed, better generalization).

### Q4: Explain XGBoost internals: second-order approximation, regularization, and column subsampling.

**A:** XGBoost optimizes gradient boosting via several innovations: (1) **Second-order approximation**: uses both first (gradient g_i) and second (Hessian h_i) derivatives of loss, enabling better step size and convergence. Objective: ∑ [g_i f_m(x_i) + (1/2) h_i f_m(x_i)²] (quadratic approximation vs. linear gradient), (2) **Regularization**: L(tree) = ∑ r_i + λ ∑ w_j² + γ T (penalizes leaf count T and weight magnitude), explicitly controlling complexity—critical for preventing overfitting, (3) **Subsampling**: row subsampling (subsample rows per tree) and column subsampling (subsample features), both reduce overfitting and improve generalization, (4) **Sparsity handling**: efficiently handles missing values and sparse data (common in real-world). Key hyperparameters: learning_rate (0.01-0.1, smaller ↔ more iterations), max_depth (3-10), subsample (0.5-1.0), colsample_bytree (0.5-1.0), lambda/gamma (regularization). Tuning strategy: coarse search on depth/learning_rate, fine-tune with subsampling and regularization. Early stopping monitors validation error and halts when not improving, preventing overfitting. XGBoost is industry-standard; understanding its internals demonstrates depth. Mention that LightGBM and CatBoost are faster alternatives with similar principles, using different approximations (histogram-based, ordered boosting).

### Q5: Explain LightGBM: leaf-wise growth, GOSS, and EFB.

**A:** LightGBM prioritizes speed and memory efficiency via: (1) **Leaf-wise growth**: builds trees by recursively splitting the leaf with maximum loss reduction, not depth-wise (level by level). This enables deeper trees with fewer iterations; trade-off: complexity is higher, pruning more critical, (2) **GOSS (Gradient-based One-Side Sampling)**: keeps instances with large gradients (large loss) and randomly samples instances with small gradients (assume they contribute less to information gain). This reduces data size per iteration while preserving accuracy; assumes gradients correlate with importance, (3) **EFB (Exclusive Feature Bundling)**: bundles mutually exclusive features (rare co-occurrences) into single features, reducing dimensionality without information loss; particularly effective for sparse data. Advantages: (1) much faster training than XGBoost (10-20x on large data), (2) lower memory footprint (GOSS, EFB reduce data), (3) handles categorical features natively (no one-hot encoding required), (4) hyperparameters similar to XGBoost (easy transition). Disadvantages: (1) less stable on small datasets (< 10k samples; GOSS sampling adds noise), (2) fewer theoretical guarantees than XGBoost, (3) different hyperparameter tuning needed (num_leaves vs. max_depth). In practice, LightGBM is preferred for large datasets (> 100k) due to speed; XGBoost for small-to-medium datasets where stability matters. In interviews, mentioning GOSS/EFB shows awareness of computational optimizations. Discuss trade-off: speed vs. stability based on data size.

### Q6: Explain CatBoost and its handling of categorical features and ordered boosting.

**A:** CatBoost specializes in tabular data with categorical features, addressing practical pain points: (1) **Native categorical support**: handles categorical features internally without one-hot encoding, using target encoding (encoding each category as its mean target value). Avoids high-dimensional sparse data that slows tree-building. Encoding is done in a fold (temporal ordering) to prevent information leakage, (2) **Ordered boosting**: during training, use only samples with indices < current sample (temporal ordering) to compute gradients. This prevents information leakage—current sample isn't used to compute its own gradient—improving generalization empirically, (3) **Symmetric trees**: same split feature/threshold at all nodes of a level, enabling computational efficiency and preventing overfitting. Advantages: (1) minimal preprocessing (categorical features passed as-is), (2) strong generalization on categorical data (ordered boosting), (3) stable—requires less hyperparameter tuning than XGBoost, (4) good default hyperparameters. Disadvantages: (1) slower training than LightGBM (ordered boosting and target encoding have overhead), (2) ordered boosting may reduce diversity on certain data, (3) less widely adopted (ecosystem smaller). When to use: datasets with heavy categorical features, little domain knowledge for feature engineering, need robust generalization. CatBoost often wins with minimal tuning. In interviews, mentioning CatBoost shows awareness of specialized optimizations; many practitioners aren't familiar with it. Discuss that default CatBoost often outperforms heavily-tuned XGBoost on categorical data.

### Q7: Explain stacking and how it differs from bagging and boosting.

**A:** Stacking (stacked generalization) combines diverse models via a meta-learner. Algorithm: (1) Split data into k folds, (2) for each fold: train base models on k-1 folds, predict on held-out fold, (3) concatenate out-of-fold predictions (meta-features), (4) train meta-learner (typically linear regression or logistic regression) on meta-features to predict target. Final prediction: meta-learner(base_model_1(x), base_model_2(x), ...). Advantages: (1) highly flexible (any base models, any meta-learner), (2) can combine diverse models (e.g., linear + tree + SVM), (3) often achieves best performance (competitions often won by stacking), (4) base models can use different features/algorithms. Disadvantages: (1) complex (many hyperparameters), (2) requires careful implementation (prevent data leakage—use out-of-fold predictions), (3) slow training (train k × M base models, M = number of base models), (4) inference involves multiple models (slower). Stacking vs. bagging: bagging uses same model type, reduces variance; stacking combines diverse models, can reduce both bias and variance. Bagging is simpler and faster. Stacking vs. boosting: boosting sequential, reduces bias; stacking parallel, more flexible. Boosting is usually better for underfitting. Use stacking: (1) competitions where every % matters, (2) heterogeneous data where different models excel on different features, (3) when computational cost is acceptable. In practice, XGBoost often suffices; stacking is for squeezing final % gains. In interviews, stacking shows advanced knowledge; mention it as a tool for maximizing performance when time/computation permits.

### Q8: Explain blending and its relationship to stacking.

**A:** Blending is a simpler variant of stacking avoiding k-fold cross-validation overhead. Algorithm: (1) Split data into train (80%), validation (20%), (2) train base models on train set, predict on validation set, (3) train meta-learner on validation predictions. Final prediction: meta-learner(base_models(x)). Advantages: (1) simpler than stacking (one split vs. k-fold), (2) faster (fewer model fits), (3) less risk of overfitting meta-learner (validation set is held out). Disadvantages: (1) wastes 20% of data (validation not used for base model training), (2) meta-learner sees only validation distribution (potential distribution shift if validation ≠ test), (3) higher variance in meta-features (smaller validation set). Blending vs. stacking: stacking uses out-of-fold predictions (all training data leveraged), lower variance, more robust; blending faster but wastes data and risks distribution shift. In competitions: stacking preferred (every data point matters), in production: blending acceptable (simpler, faster). Practical comparison: stacking O(k × M) training, blending O(M + 1) training. In interviews, mention blending as a practical trade-off when time is limited. Note that stacking dominates competitions due to better utilization of training data.

### Q9: Explain model diversity and why it's critical for ensemble success.

**A:** Ensemble strength depends on individual model diversity—if all models make the same errors, averaging doesn't help. Consider extreme: if two models are identical, ensemble error = individual error (no benefit). If models are independent (random errors), ensemble variance is σ²/M (full variance reduction). Mathematically, ensemble variance is Var(ensemble) = (1/M) E[Var_i] + (1 - 1/M) E[Cov(error_i, error_j)]. Variance reduction depends on covariance—negative or zero covariances are ideal. Strategies to ensure diversity: (1) **Different algorithms**: combine trees, linear models, neural nets; different inductive biases capture different patterns, (2) **Subsampling**: bagging creates diversity via bootstrap samples; each model sees different data, (3) **Feature subsets**: column subsampling (XGBoost) or random subspaces; models focus on different features, (4) **Hyperparameter variation**: shallow vs. deep trees, different regularization; explicit diversity lever, (5) **Sequential training** (boosting): each model focuses on previous model's errors, implicitly creating diversity. Empirically, diverse weak learners (individual error < 60%, negatively correlated) yield excellent ensembles. Homogeneous ensembles (same algorithm, hyperparameters) have limited diversity—bagging helps via data variation, but mixing algorithms (stacking) is stronger. Red flag: if all base models are highly correlated (e.g., all deep neural nets with similar architecture), ensemble gains are minimal. In interviews, discussing diversity as the key driver of ensemble effectiveness shows deep understanding beyond "just average models."

### Q10: Explain ensemble pruning and when/why you'd reduce ensemble size.

**A:** Ensemble pruning removes base models (or predictions) to improve efficiency without sacrificing—sometimes even improving—accuracy. Motivation: (1) full ensemble is expensive (inference time, memory), (2) some models are redundant (correlated errors), (3) removing them can improve generalization (avoid over-reliance on noisy models). Pruning strategies: (1) **Greedy removal**: iteratively remove model with lowest impact on ensemble accuracy; continue until accuracy degradation is acceptable, (2) **Threshold-based**: remove models with error above threshold, (3) **Diversity-based**: keep models with highest diversity (negative correlation); remove redundant ones, (4) **Optimization**: formulate as subset selection problem, find k-subset maximizing accuracy + diversity. Implementation: (1) compute out-of-fold predictions on validation set, (2) evaluate subsets via cross-validation, (3) select the subset with best accuracy per model count trade-off. Trade-off: larger ensemble ↔ higher accuracy but slower inference. Example: 100-model ensemble takes 100x inference time; pruning to 20 models might reduce accuracy by 0.5% but cut inference by 80%. In production: pruning is valuable (latency constraints), in competitions: no pruning (maximize accuracy). Surprisingly, pruning sometimes improves test accuracy—removes models overfit to training distribution. In interviews, pruning shows awareness that "more is not always better" and practical concern for inference speed. Mention that this is underexplored in practice—many practitioners use all trained models without questioning.

### Q11: Explain dropout as an ensemble method and its connection to model regularization.

**A:** Dropout is a regularization technique that can be interpreted as ensemble learning at test time. During training: at each forward pass, randomly set neurons to zero with probability p (typically 0.5), effectively training a sub-network. At test time: use all neurons but scale by (1-p) to account for training expectations. Ensemble interpretation: each dropout sample is a different sub-network; during training with dropout, you're implicitly training an ensemble of 2^n sub-networks (where n = number of neurons). At test time, the scaled network approximates averaging over all sub-networks. Advantages of dropout: (1) **regularization**: prevents co-adaptation of neurons, (2) **implicit ensemble**: approximates ensemble averaging with single model inference (no retraining needed), (3) **adaptive**: learned implicit ensembles, (4) computational efficiency: training cost same as no dropout. Disadvantages: (1) increases training time (stochastic sampling), (2) requires careful tuning (dropout rate p), (3) less interpretable than explicit ensembles. Empirically, dropout is among the best regularization techniques for neural networks. Connection to explicit ensembles: dropout is "thin" ensemble (parameter sharing across sub-networks), while explicit ensembles are "fat" (independent models). Dropout combines efficiency of single model with ensemble benefits. In interviews, explaining dropout as ensemble method shows theoretical understanding beyond standard "regularization" explanation. Mention that modern interpretations (variational inference) further justify dropout's effectiveness.

### Q12: What are common pitfalls in ensemble learning and how do you avoid them?

**A:** (1) **Data leakage in stacking**: using same data for base model training and meta-learner training causes overfitting. Solution: use out-of-fold (CV) or blending (held-out validation). (2) **Over-regularization in boosting**: regularizing too heavily (low learning rate, early stopping too early) removes signal. Solution: tune regularization via validation, not by hand. (3) **Ignoring base model quality**: garbage in, garbage out—weak base models (random predictions) don't help. Solution: ensure base models > 50% accuracy (or zero error for regression). (4) **Insufficient diversity**: using similar models (e.g., all XGBoost with different seeds) provides limited benefit. Solution: mix algorithm types (linear, tree, neural net). (5) **Model overfitting despite ensemble**: complex base models or stacking meta-learner can overfit. Solution: regularize base models, use simple meta-learner (linear), monitor validation performance. (6) **Inference latency underestimated**: ensemble inference is M times slower. Solution: profile early, prune if latency is critical. (7) **Hyperparameter tuning on full ensemble**: ignores individual model quality. Solution: tune base models individually first, then ensemble hyperparameters. (8) **Imbalanced ensemble**: models with different quality—averaging weights them equally, but better models deserve higher weight. Solution: weighted averaging based on validation accuracy. In interviews, acknowledging these pitfalls shows practical experience. Discuss that stacking is powerful but requires careful engineering; boosting can backfire if over-regularized. Strong answer: "Ensembles are powerful but introduce complexity; understand your base models first, ensure diversity, validate carefully."

### Q13: When would you use voting classifiers and what's the difference between hard and soft voting?

**A:** Voting classifiers combine predictions from multiple independent models. Hard voting: assign class by majority vote (most common prediction). Soft voting: average predicted probabilities, then assign class with highest probability. Hard voting: ŷ = argmax_k ∑_i [ŷ_i(x) = k] (count votes). Soft voting: ŷ = argmax_k (1/M) ∑_i P(y=k | x_i). Soft voting generally performs better because it leverages probability estimates (richer information than hard class predictions). However, soft voting requires all base models to output probabilities (some don't naturally—SVM with default settings). Hard voting is robust (any base model works), soft voting is more precise. Example: two logistic regressions (output probability 0.51, 0.52 for class 1) hard-vote as class 1 (majority), soft-vote as class 1 (average probability 0.515). Voting with equal weights assumes equal base model quality; better approach: weight by validation accuracy. Weighted soft voting: ŷ = argmax_k ∑_i w_i P(y=k | x_i). Voting is simpler than stacking (no meta-learner fitting required), faster inference (simple aggregation), but less flexible (can't learn interactions). Use voting: (1) independent models (from different sources/teams), (2) need simplicity/speed, (3) base models are well-validated. Use stacking: (1) want to optimize ensemble, (2) computational cost acceptable, (3) meta-learner can capture model strengths. In interviews, mentioning soft vs. hard voting and weighted variants shows practical awareness—many practitioners use unweighted hard voting, unaware of alternatives.

### Q14: How do you evaluate ensemble performance and avoid overfitting?

**A:** Evaluation best practices: (1) **Separate test set**: always reserve held-out test data never seen during training/tuning; ensemble metrics on test set only, (2) **Nested cross-validation**: outer CV for ensemble evaluation, inner CV for base model tuning; prevents overfitting to outer fold, (3) **Early stopping**: for boosting, monitor validation error, halt when not improving; prevents overfitting, (4) **Out-of-fold evaluation**: for stacking, use CV-generated predictions (never in-fold), ensuring honest meta-learner performance. Overfitting risks: (1) meta-learner overfitting to training meta-features—solution: regularize (L2 if linear), use validation-based early stopping, (2) base model overfitting—solution: regularize individually (depth, L2, dropout), (3) hyperparameter tuning overfitting—solution: use proper CV, separate test set for final evaluation. Red flags: (1) training ensemble accuracy >> validation accuracy (overfitting), (2) ensemble outperforms base models by tiny margin (maybe noise), (3) ensemble performance improves indefinitely with model count (indicates overfitting). Validation strategy: (1) split into train/validation/test, (2) train base models on train, (3) generate meta-features on validation (or CV on train), (4) evaluate on test only, (5) tune hyperparameters on validation/CV, not test. Never touch test set until final evaluation. In interviews, emphasizing careful evaluation and avoiding overfitting demonstrates rigor. Mention that many practitioners overlook proper CV for ensemble tuning, leading to overoptimistic results.

### Q15: When is a single model preferable to an ensemble, and what are computational trade-offs?

**A:** Ensembles improve accuracy but add complexity and latency. Use single model when: (1) **Inference latency critical**: ensemble inference is M times slower (inference cost dominates, e.g., real-time recommendations); single model or pruned ensemble needed, (2) **Simplicity required**: deployment, maintenance, interpretability easier with single model, (3) **Single model is strong enough**: if XGBoost achieves 95% accuracy and ensemble achieves 95.2%, improvement may not justify 5-10x latency increase, (4) **Computational resources limited**: training ensembles requires more memory/storage, (5) **Interpretability required**: regulatory settings (finance, healthcare) prefer single interpretable model over black-box ensemble. Use ensemble when: (1) **Accuracy critical**: medical diagnosis, fraud detection where every % matters, (2) **Inference latency acceptable**: batch processing, offline analysis, (3) **Diverse base models available**: different algorithms/features provide genuine diversity, (4) **Competition/research**: maximize accuracy at any cost. Practical guidance: (1) always train baseline (single model), (2) if ensemble gain >> latency cost, use ensemble, (3) in production, profile inference latency first—choose based on SLA (service-level agreement), (4) consider hybrid: single fast model + ensemble for final refinement. Cost-benefit analysis: gain in accuracy / latency multiplier. Example: 1% accuracy gain with 5x latency cost is poor trade-off for most applications. In interviews, discussing this trade-off shows maturity—many practitioners reflexively ensemble, unaware of hidden costs. Strong answer: "Start with single well-tuned model; only ensemble if accuracy gap is significant and latency allows."

## Interview Cheatsheet

**Key Terms:**

- **Bias-Variance Decomposition:** MSE = Bias² + Variance + Noise; bagging reduces variance, boosting reduces bias.
- **Bagging:** Bootstrap Aggregating; trains parallel models on bootstrap samples; reduces variance via averaging.
- **Bootstrap Sample:** Sampling with replacement; ~63% unique, ~37% duplicates; creates data diversity.
- **Boosting:** Sequential training on residuals/misclassified samples; reduces bias via iterative improvement.
- **AdaBoost:** Reweights samples, increases weights on misclassified; weak learners → strong learner.
- **Gradient Boosting:** Fits pseudo-residuals (loss gradients); flexible loss function, often best performance.
- **XGBoost:** Regularized gradient boosting; second-order approximation, subsampling, complexity penalties.
- **LightGBM:** Fast gradient boosting; GOSS (sampling), EFB (feature bundling), leaf-wise growth.
- **CatBoost:** Categorical-focused boosting; target encoding, ordered boosting, no one-hot encoding needed.
- **Stacking:** Combines diverse models via meta-learner; flexible, complex, often best accuracy.
- **Blending:** Simpler stacking variant; one train-val split, faster but wastes data.
- **Model Diversity:** Ensemble strength depends on base model differences; diverse errors cancel via averaging.
- **Soft Voting:** Average predicted probabilities; better than hard voting (majority vote).
- **Hard Voting:** Majority class vote; simpler, but ignores probability magnitudes.
- **Dropout:** Training regularization approximating ensemble; set neurons to zero with probability p.
- **Ensemble Pruning:** Remove redundant models; improves inference speed, sometimes generalization.

**Rapid-Fire Q&A:**

- **Q:** Bagging or boosting to reduce overfitting? **A:** Bagging—reduces variance of high-variance models; boosting reduces bias (for underfitting).
- **Q:** When does bagging help most? **A:** High-variance base models (deep trees); low-variance models (linear) benefit little.
- **Q:** XGBoost vs. random forest both ensemble—which faster? **A:** Random forest faster inference (parallel); XGBoost slower training but often better accuracy; choice depends on requirements.
- **Q:** Ensemble always better than single model? **A:** No—if single model is strong, ensemble gain may not justify latency/complexity; always compare cost-benefit.
- **Q:** What breaks ensemble averaging? **A:** High model correlation (similar errors don't cancel); ensure diversity via algorithms, subsampling.
- **Q:** Stacking or boosting? **A:** Boosting reduces bias (sequential, specific goal); stacking more flexible (any models, meta-learner); boosting usually simpler.
- **Q:** Data leakage in stacking: what? **A:** Using same data for base training and meta-learning; solution: out-of-fold (CV) or validation set.
- **Q:** Soft or hard voting? **A:** Soft voting usually better (leverages probabilities); hard voting simpler, works with any model.
- **Q:** Early stopping in boosting why? **A:** Prevents overfitting; validation error may increase while training error decreases; halt at valley.
- **Q:** Dropout interpretation as ensemble? **A:** Training sub-networks; at test time, scaled network approximates ensemble average.
- **Q:** Ensemble inference latency? **A:** M times slower (M models); profile early, prune if SLA violated.
- **Q:** GOSS in LightGBM: purpose? **A:** Gradient-based sampling—keep high-gradient (large error) samples, randomly subsample low-gradient; speeds up.
- **Q:** Weighted voting vs. equal weights? **A:** Weighted better (high-accuracy models get higher influence); equal weights assume equal quality.
- **Q:** When prune ensemble? **A:** Production (latency critical); may also improve generalization (removes noisy models).
- **Q:** AdaBoost sensitive to? **A:** Noisy data (noise reweighted as hard samples, exponentially amplified); requires > 50% weak learner accuracy.
