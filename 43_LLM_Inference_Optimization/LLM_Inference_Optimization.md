# LLM Inference Optimization

📺 **Video Lecture:** https://youtu.be/W8YlpDD9kuA

## Interview Anchor
- **Quantization:** Reducing numerical precision (INT8, INT4) to decrease model size and latency without major quality loss.
- **KV-Cache Optimization:** Storing pre-computed key-value pairs to avoid recomputation during generation, with paged attention enabling efficient memory use.
- **Speculative Decoding:** Using a fast draft model to generate candidate tokens, then verifying with the target model, parallelizing generation for speedup.

## Key Concepts Overview
LLM inference optimization is critical because language models are expensive to deploy: a 70B-parameter model requires 140GB of GPU memory (8-byte floats) and generates one token at a time, creating latency bottlenecks. A single query on a naive implementation might take 30+ seconds, making real-time applications infeasible. Optimization techniques reduce cost (hardware, electricity), latency (time-to-first-token, inter-token latency), and enable deployment on smaller hardware (edge devices, consumer GPUs). The challenge is balancing competing objectives: quantization reduces memory but hurts quality; batching improves throughput but increases latency; speculative decoding speeds inference but requires training or retrieving draft models. Understanding these tradeoffs and modern serving frameworks is essential for deploying production LLMs, especially in cost-conscious environments where each dollar of compute must be justified.

### Q1: Why is inference optimization critical for LLMs? Explain the cost-latency-quality tradeoff.

**A:** LLM inference is expensive because models are massive (7B to 405B parameters), generate tokens autoregressively (one at a time, sequentially), and consume significant GPU memory and compute per token. A naive implementation of a 70B model requires ~140GB memory (each parameter is 2 bytes in FP16), each token takes ~100ms on an H100 GPU, and generating a 500-token response takes 50 seconds—unacceptable for real-time applications. The primary costs are: (1) memory (scales with model size; 405B parameter model doesn't fit on single GPU), (2) latency per token (user experience depends on time-to-first-token and inter-token latency), (3) compute (proportional to tokens × model size, expensive at scale). Optimization techniques introduce tradeoffs: quantization (INT4, INT8) reduces memory by 75-87.5% and speeds compute, but degrades quality (especially reasoning tasks suffer more than summarization). Batching increases throughput (tokens/second) by processing multiple requests simultaneously, but increases per-request latency (single request slower when batched than run alone). Speculative decoding speeds inference but introduces slight quality degradation if draft model is weak. Cost-latency-quality optimization requires: (1) understanding your bottleneck (memory-bound vs. compute-bound), (2) choosing optimizations matching your constraints (mobile device needs extreme quantization; data center with GPUs can prioritize latency), (3) careful evaluation on downstream tasks (optimize for end-user quality, not just benchmarks). In practice, most deployments combine multiple techniques: quantize to fit memory, use KV-cache optimization to reduce latency, batch requests to improve throughput, and measure actual costs on representative workloads.

### Q2: What is quantization? Compare INT8, INT4, and GPTQ. What are the tradeoffs?

**A:** Quantization reduces numerical precision, storing numbers in fewer bits (INT8 = 8-bit integers, INT4 = 4-bit integers) instead of 32-bit floats (FP32) or 16-bit floats (FP16). For example, FP32 uses 32 bits per parameter; INT8 uses 8 bits (96.9% reduction); INT4 uses 4 bits (99.2% reduction). Quantization works because neural networks are robust to noise—modest precision loss has minimal accuracy impact. INT8 quantization: (1) computes per-channel or per-layer scaling factors (min/max values), (2) linearly maps FP32 → INT8 range, (3) performs matrix multiplications in INT8 (much faster than FP32), (4) dequantizes output back to FP32. INT8 incurs 10-20% accuracy loss on average, works well for most tasks, supported by standard libraries (bitsandbytes). INT4 quantization is more aggressive: 50-80% accuracy loss on reasoning tasks, but acceptable for semantic tasks (summarization, classification). GPTQ (Generative Pre-Trained Transformer Quantization) improves INT4 by carefully selecting which weights to quantize least aggressively: (1) computes Hessian of loss (curvature, sensitivity to weight changes), (2) quantizes weights with low curvature more aggressively (lower precision), high-curvature weights less aggressively (higher precision), (3) uses layer-by-layer approach, only requiring calibration data and modest GPU memory. GPTQ achieves excellent INT4 accuracy (similar to INT8) because it respects model sensitivity. Comparison: INT8 is simple, fast, supported everywhere, minimal accuracy loss; INT4 saves more memory, requires techniques like GPTQ or AWQ (Activation-Aware Quantization Weighting) for acceptable accuracy, slower inference than INT8 (bit-packing overhead). Tradeoffs: INT8 best for most applications; INT4 when memory is critical (edge deployment, very large models); FP16 when accuracy is paramount. Practical insight: quantize in the order INT8 → INT4 → FP8, evaluating quality at each step using downstream metrics.

### Q3: What is knowledge distillation? How is it used for LLM optimization?

**A:** Knowledge distillation trains a smaller "student" model to mimic a larger "teacher" model, enabling deployment of smaller, faster models with quality closer to the original. The teacher (e.g., GPT-3.5) produces soft targets: instead of hard labels (e.g., probability 1.0 for correct token, 0 for others), use the full probability distribution from the teacher, preserving information about which wrong answers the teacher considers plausible. Student loss combines: (1) distillation loss (KL divergence between student and teacher logits, with temperature τ >> 1 to soften distributions), (2) task loss (standard cross-entropy on ground truth). Distillation works because the teacher's soft targets contain more information than hard labels—if teacher assigns 0.3 probability to a plausible-but-wrong token, the student learns this is an understandable mistake. For LLMs: (1) generate responses from a teacher LLM (GPT-4) on a dataset, (2) train a student LLM (e.g., 7B parameters) to match teacher outputs, (3) student becomes cheaper to deploy while maintaining quality. Advantages: smaller models (70% fewer parameters possible), faster inference (3-5x speedup), maintains semantics (student learns reasoning patterns from teacher). Limitations: (1) requires running expensive teacher many times (data generation cost), (2) student cannot exceed teacher quality (upper bound on performance), (3) choice of dataset and temperature critically affects results, (4) some tasks require teacher-specific knowledge (ChatGPT's instruction-following can't be fully distilled). In practice, combine distillation with quantization: distill to 7B model, then quantize to INT8/INT4, achieving both smaller model and lower precision. The frontier is online distillation (interleaving student training with teacher inference) reducing data generation cost. Knowledge distillation is particularly useful for chat models where latency and cost matter; deploying teacher directly is expensive, but smaller distilled student can serve most queries.

### Q4: What is model pruning? Explain structured vs. unstructured pruning and when to use each.

**A:** Pruning removes weights or neurons with small magnitudes, assuming they contribute minimally to predictions. Unstructured pruning removes individual weights (e.g., zeroing out 50% of weights with smallest absolute values), creating sparse weight matrices with many zeros. Structured pruning removes entire filters, heads, or layers (e.g., removing entire attention heads from a transformer), maintaining dense matrices that are hardware-friendly. Unstructured pruning: (1) apply sparsity mask (set small weights to zero), (2) fine-tune on calibration data to recover accuracy, (3) inference requires specialized sparse matrix operations (not all hardware accelerates sparse operations, limiting speedup). Structured pruning: (1) identify and remove entire components (attention heads, layers), (2) fine-tune, (3) inference uses standard dense operations, immediately faster on any hardware. For LLMs: structured pruning is preferred because transformers have easily-identifiable components (heads, layers) and unstructured sparsity doesn't accelerate on GPUs/TPUs. Example: a 175B model might prune 20% of layers + 30% of attention heads, achieving 40% reduction with 2-3x speedup and modest accuracy loss. Head pruning is particularly effective: many attention heads are redundant; pruning low-importance heads (by attention gradient or contribution to loss) maintains performance. Pruning requires: (1) identifying which weights are important (magnitude, gradient-based, Fisher information, Hessian-based), (2) removing unimportant weights, (3) fine-tuning to recover accuracy, (4) iterative pruning (prune more, fine-tune, repeat). Tradeoffs: unstructured pruning achieves more aggressive reduction (up to 90%) but requires specialized hardware; structured pruning is hardware-friendly but achieves less reduction (typically 30-50%). For production LLMs, structured layer/head pruning is most practical. Modern trend: combine pruning with distillation (distill to smaller model, prune unimportant components, quantize).

### Q5: Explain speculative decoding. How does it speed inference? What are the limitations?

**A:** Speculative decoding addresses the bottleneck that LLM generation is memory-bound: computing the next token requires loading the entire model (weights + activations), while actual compute is minimal relative to memory access. Speculative decoding uses a draft model to generate K candidate tokens in parallel (fast, uses small model), then the target model verifies all K tokens in one forward pass, accepting correct tokens and re-sampling from the first incorrect position. This parallelization enables speedup: instead of K sequential slow passes (generate K tokens), you do 1 fast pass (draft) + 1 slow pass (verify K tokens), yielding near 1 + 1/K ≈ 2x speedup for K=10. Specifically: (1) draft model (small, fast—could be 7B while target is 70B) generates tokens t₁, t₂, ... tₖ, (2) target model computes P_target(t₁), P_target(t₂), ..., P_target(tₖ) in a single batch, (3) verify by comparing target probability to draft probability—if P_target(tᵢ) > draft probability (with rejection sampling adjustment), accept tᵢ; else, re-sample from target distribution and stop. The key insight: rejection sampling ensures output distribution matches target model exactly, no quality loss. Advantages: significant speedup (1.5-3x depending on draft model quality), no quality degradation (output distribution is exact), compatible with other optimizations (quantization, KV-cache). Limitations: (1) requires a good draft model (weak draft model means low verification rate, minimal speedup; draft model should be ~10-20x faster than target), (2) fine-tuning target model invalidates draft model (need to re-train draft), (3) draft model adds complexity (training, storage), (4) parallelization only works when K is small (verifying 100 tokens becomes slow; typical K=5-20). Practical deployment: use existing model as draft (e.g., Llama-7B drafts for Llama-70B), or train lightweight distilled version. Speculative decoding is increasingly used in production (Google Gemini, vLLM supports it) because speedup is substantial and implementation is clean.

### Q6: What is KV-cache optimization? Explain paged attention and vLLM.

**A:** KV-cache stores pre-computed key and value embeddings from previous tokens, avoiding recomputation during generation. Without KV-cache: generating token tᵢ requires running attention on all tokens t₁...t_{i-1}, which is O(i) cost—expensive for long sequences. With KV-cache: cache K and V matrices from previous forwards, so computing attention for token tᵢ only requires multiplying new query with cached K,V, reducing O(i) to O(1) per token. Memory tradeoff: KV-cache is large (for a 70B model generating a 2000-token sequence, KV-cache is ~280GB in FP16), often dominating total memory usage and limiting batch size (multiple requests compete for KV-cache memory). Paged attention (introduced in vLLM) optimizes KV-cache management by: (1) storing KV-cache in fixed-size blocks (pages) rather than contiguous arrays, (2) using virtual memory style paging (request 1's cache in pages {0,2}, request 2's cache in pages {1,3}), (3) enabling efficient memory reuse—if requests share a prefix (common in batch processing), share cached pages rather than duplicate. This reduces memory fragmentation and enables higher batch sizes. vLLM (vector LLM) is a serving engine implementing paged attention, enabling near-GPU-memory-limited throughput: (1) manages request batching to maximize KV-cache utilization, (2) prefill phase (initial prompt processing) is batched across requests, generation phase streams tokens, (3) reorders requests dynamically to keep GPU busy. Practical impact: without optimization, serving 70B model processes ~1-5 requests simultaneously; vLLM with paged attention enables 10-50 requests simultaneously on same GPU (16x improvement). Implementation details: (1) block size (typically 16-128 tokens) balances page overhead vs. granularity, (2) block mapping tracks which request owns which pages, (3) attention kernel modified to support non-contiguous KV locations. Trade-offs: paged attention adds complexity (block management), marginal quality issues if block boundaries coincide with important token boundaries (rare). Modern LLM serving (vLLM, TGI, TensorRT-LLM) all implement variants of paged attention.

### Q7: Explain continuous batching and how it improves serving throughput compared to static batching.

**A:** Continuous batching (dynamic batching) processes requests as they arrive, removing completed requests and adding new ones without waiting for a fixed batch to fill. Static batching waits for a full batch (e.g., 32 requests) before processing, causing high latency: if batch size is 32 and only 1 request arrives, it waits for 31 more. Continuous batching: (1) request arrives, added to batch, (2) GPU processes all requests one token generation step, (3) if request finishes, remove it immediately, (4) add new arriving requests, (5) repeat. This maximizes GPU utilization and minimizes latency. Example: requests have lengths [10, 50, 30, 20] tokens to generate. Static batching (batch size 4): waits for all 4 requests, processes 50 steps (longest request), total latency ~50 steps. Continuous batching: request 1 finishes at step 10 (immediately dequeued), step 20 request 4 finishes, new requests added, maximizing parallelism without waiting for stragglers. Throughput improves because GPU processes varying-length requests efficiently instead of padding short requests to batch maximum. Implementation: (1) tokenize requests on arrival, add to queue, (2) each token generation iteration, process all requests with batch size = current queue length, (3) decrement length counter for each request, (4) remove completed requests. With paged attention (KV-cache efficiently manages variable request lengths), continuous batching achieves near-optimal throughput. Trade-offs: more complex scheduling logic, potentially higher CPU overhead (managing queue), slight increase in inter-token latency (varying batch sizes). In practice: vLLM, TGI, and modern serving frameworks all implement continuous batching by default. Measured impact: 2-4x throughput improvement over static batching with similar latency.

### Q8: What is tensor parallelism vs. pipeline parallelism? When do you use each for serving?

**A:** Both techniques split a model across multiple GPUs for inference. Tensor parallelism (TP) splits tensors horizontally across GPUs: matrix multiplication A @ B is split as (A₁ @ B₁) + (A₂ @ B₂), where each GPU computes a portion. For a transformer layer with weight matrix W (size [4096, 4096]), TP=2 splits W into [2048, 4096] on GPU 1 and [2048, 4096] on GPU 2, each computing half the output. Requires AllReduce communication after each layer to combine partial results. Pipeline parallelism (PP) splits layers vertically: GPU 1 runs layers 0-20, GPU 2 runs layers 21-40, etc. Requests flow through pipeline: GPU 1 processes token 1 in layers 0-20, passes to GPU 2; meanwhile GPU 1 processes token 2. Requires point-to-point communication only between adjacent GPUs, less communication overhead. Tensor parallelism: (1) communication per token is high (AllReduce after each layer), (2) works well with small number of GPUs (2-4), (3) latency increases with degree (more splits = more communication), (4) good for latency-sensitive serving (e.g., chatbots). Pipeline parallelism: (1) communication per token is low (point-to-point), (2) scales to many GPUs, (3) introduces pipeline bubble (GPUs idle waiting for previous GPU), (4) requires careful scheduling (1F1B—one forward, one backward—reduces bubble to ~30%), (5) better for throughput-optimized serving. In practice: (1) single GPU: no parallelism needed; (2) 2-4 GPUs: tensor parallelism (low communication, simple), or combined TP+PP; (3) 8+ GPUs: pipeline parallelism, possibly with tensor parallelism within each PP stage. Example: 405B model on 8xH100 GPUs might use TP=2, PP=4 (4 stages of 2 GPUs each, each stage uses tensor parallelism on 2 GPUs). Choice depends on model size, GPU count, latency requirements, and batch size (larger batches favor PP to amortize communication overhead).

### Q9: What is FlashAttention and memory-efficient attention? How does it improve speed and memory?

**A:** Standard attention computes similarity matrix Q @ K^T (size [seq_len, seq_len]) for all positions, creating O(seq_len²) memory and compute. For a 2K-token sequence with 70B model, the attention matrix is 2K × 2K = 4M elements × 2 bytes = 8MB per attention head, summing to ~100MB+ for all heads—a significant memory bottleneck limiting batch size and maximum sequence length. FlashAttention optimizes attention by: (1) tiling (dividing matrices into blocks), (2) computing attention on blocks, (3) using lower precision (FP16) for intermediate computations while maintaining FP32 for numerical stability, (4) avoiding materializing the full QK^T matrix, instead computing row-wise softmax in a single pass. The algorithm: (1) divide Q, K, V into blocks, (2) for each block of Q, compute attention only against relevant K, V blocks, (3) use online softmax (Welford's algorithm) to normalize without materializing full matrix. Implementation details: (1) careful IO management (minimizing memory transfers between GPU global/shared memory), (2) hand-optimized CUDA kernels (standard PyTorch implementation is slow because attention is memory-bound, not compute-bound). Speedup: 2-3x faster, 10-20x less memory. FlashAttention-2 (and variants like FlashAttention-3) further optimize: (1) better IO scheduling, (2) multi-query attention support (KV heads < Q heads, reducing KV memory), (3) block-sparse attention (sparsity patterns reduce computation). Practical impact: enables 2-3x larger batch sizes, or 2-3x longer sequences on same hardware. Trade-offs: implementation complexity (requires careful CUDA optimization), limited adoption (only in modern libraries like vLLM, torch.compile), some numerical stability concerns in extreme cases. Modern serving frameworks (vLLM, TGI) enable FlashAttention by default. For practitioners: ensure using frameworks supporting FlashAttention; enabling it is low-effort, high-impact optimization.

### Q10: What inference frameworks exist (vLLM, TGI, TensorRT-LLM, Triton)? Compare their strengths.

**A:** Multiple specialized LLM serving frameworks address different use cases. vLLM (Berkeley) focuses on throughput: implements paged attention, continuous batching, tensor parallelism, optimized for serving many concurrent requests. Strengths: simple Python API, integrated with HuggingFace (drop-in replacement), excellent throughput (16-50x faster than baseline). Limitations: primarily GPU-focused, less mature for edge/CPU deployment. TGI (Text Generation Inference, HuggingFace): similar to vLLM, emphasizes ease-of-use and compatibility. Strengths: Docker container simplifies deployment, supports multiple model formats (safetensors, ONNX), production-ready (used at HuggingFace.co). Limitations: less throughput-optimized than vLLM, heavier Docker image. TensorRT-LLM (NVIDIA): compiles LLMs to optimized CUDA kernels using TensorRT. Strengths: fastest performance (hand-optimized kernels, fused operations), precise control (plugin architecture for custom kernels), best for latency-sensitive applications. Limitations: steeper learning curve, NVIDIA-specific (requires NVIDIA hardware). Triton (NVIDIA): general-purpose serving framework supporting multiple model types, not LLM-specific. Strengths: unified API across model types (vision, NLP, diffusion), flexible (can mix models), Python backends enable custom logic. Limitations: requires more manual tuning, less LLM-optimized. Comparison table: vLLM excels in developer experience and throughput; TensorRT-LLM in absolute performance; TGI in deployment simplicity; Triton in flexibility. Choice depends on: (1) throughput priority → vLLM, (2) latency priority → TensorRT-LLM, (3) ease of deployment → TGI, (4) multi-model serving → Triton. Modern trend: start with vLLM for prototyping, migrate to TensorRT-LLM for production latency-critical applications, use TGI for quick deployments. ONNX Runtime is also emerging as alternative for CPU/edge deployment, with model quantization. Most deployments use one of vLLM, TGI, or TensorRT-LLM depending on infrastructure.

### Q11: What hardware considerations matter for LLM inference? Compare GPU vs. CPU, A100 vs. H100, TPU.

**A:** LLM inference hardware choice directly impacts cost and performance. GPUs dominate because inference is memory-bandwidth-bound (high data movement relative to compute), and GPUs have high memory bandwidth. A100 (80GB HBM): released 2020, 1.9TB/s bandwidth, sufficient for 70B model in FP16, good for batch inference. H100 (141GB HBM): released 2023, 3.4TB/s bandwidth (80% more), native INT8 support, enables larger batches and longer sequences, significantly faster (~1.5-2x on latency benchmarks). H100 cost is ~2x A100, ROI depends on workload (throughput-bound workloads favor H100; latency-critical may justify cost). TPU (Google): optimized for ML, lower cost than H100, comparable performance, but ecosystem is smaller (less mature software, less community support). For cost-sensitive inference, A100 or H100 beats CPU by 100x on latency. CPU inference: 1-2% of GPU latency for same model, extremely slow for LLMs (a 70B model on CPU takes minutes per token). CPU is useful for: (1) low-volume inference (don't need high throughput), (2) edge devices (phones, IoT), (3) cost-constrained (CPU is free if already available), (4) custom logic requiring CPU (preprocessing, post-processing). Mixed approach: use GPU for LLM inference, CPU for pre/post-processing. Memory bandwidth: (1) FP32 (4 bytes per param) requires 280GB/s bandwidth for 70B model at 1 token/second, (2) FP16 (2 bytes) requires 140GB/s, (3) INT8 requires 70GB/s. H100's 3.4TB/s supports batch size ~10-20 for 70B model; A100 supports ~5-10. Quantization dramatically improves memory efficiency: INT4 reduces memory requirements 4x, enabling larger batches or multiple models on single GPU. Practical deployment: (1) small budget: quantized model on A100, (2) medium: H100, (3) large-scale: H100 clusters with TP/PP, (4) edge: quantized model on mobile CPU/GPU. Cost-per-inference is critical metric: H100 costs 100x CPU but provides 1000x speedup, excellent ROI for production inference.

### Q12: What are token-level latency metrics? How do you optimize time-to-first-token vs. inter-token latency?

**A:** Token-level latency has two components: (1) Time-to-First-Token (TTFT): latency from request arrival to receiving the first generated token, dominated by prefill (processing input prompt) and attention initialization, (2) inter-token latency (ITL): time between successive generated tokens, dominated by decoding (single-token generation). TTFT is critical for user experience (perceived responsiveness), ITL determines generation speed (words per second). For a 100-token input, 100-token output: TTFT = time to process 100 input tokens (might be 500ms batch processing), ITL = time per output token (might be 50ms, yielding 20 tokens/sec = 5 seconds for 100 tokens). Optimizing TTFT: (1) batch prefill (process multiple requests' prefills simultaneously—batch size affects throughput), (2) shorter input sequences (prompt engineering), (3) prefix caching (if many users ask similar questions, cache prefix computations), (4) reduce model size (smaller models have lower prefill latency). Optimizing ITL: (1) KV-cache reduces ITL since decoding is fast (single token attention), (2) quantization speeds decoding, (3) speculative decoding generates multiple tokens per slow forward pass, (4) batch generation (generate for multiple requests simultaneously—increases ITL slightly but improves overall throughput). Trade-off: batching reduces TTFT (amortizes prefill cost) but increases ITL (slower per-token generation). For interactive applications (chatbots), prioritize low TTFT + low ITL = small batches, fast inference. For throughput applications (bulk text generation), optimize total time = longer batches reduce TTFT cost per request, ITL is less critical. Practical metrics: (1) p50/p99 latency (50th/99th percentile) matter more than average—one slow request ruins user experience, (2) measures should separate TTFT and ITL, (3) latency should include queueing time (total user-perceived latency). Modern serving frameworks report these separately, enabling informed optimization choices.

### Q13: How do you optimize inference cost at scale? Discuss spot instances, model routing, cascading.

**A:** Cost optimization at scale requires multi-dimensional approach. Spot instances (AWS, GCP): reserved instances that can be interrupted (cost 70-90% less), for fault-tolerant workloads (batch inference, caching). Use spot for non-latency-sensitive workloads, reserve on-demand instances only for latency-critical paths. Model routing: different requests have different characteristics (input length, output length, quality requirements)—route requests to appropriately-sized models. Example: route simple queries (classification) to 7B model, complex queries to 70B model, saving cost by not using large model unnecessarily. Requires: (1) request classifier (detect if query is simple or complex), (2) quality evaluation (ensure routed model achieves acceptable accuracy), (3) fallback (if 7B fails, escalate to 70B). Cascading: arrange models by increasing capability, trying smallest first. Example: (1) 7B model generates response, (2) if confidence is high, return; if uncertain, try 13B; if still uncertain, try 70B. This reduces average cost by serving most requests with cheap model while maintaining quality. Optimization strategies: (1) request batching (wait for multiple requests to batch, amortizing overhead), (2) caching (store frequent request results, return without inference), (3) prefix caching (requests with common prefix share computation), (4) quantization (reduce per-request cost), (5) request filtering (decline low-margin requests), (6) time-of-use pricing (inference during off-peak is cheaper). Hardware multiplexing: run multiple quantized models on single GPU if memory allows (4x INT8 models on single GPU with vLLM). Example: 70B model at 1 token/sec costs $0.02 per 100 tokens; 7B model costs $0.002; routing 80% to 7B + 20% to 70B yields average $0.0056 per 100 tokens = 72% cost savings. Practical deployment: (1) measure cost per request + latency/quality tradeoff, (2) implement model routing based on request complexity (LLM-as-judge for classification), (3) monitor cost and quality continuously, (4) adjust routing thresholds based on economics. Cost-aware ML is increasingly important; a 50% cost reduction can make inference viable at scale.

### Q14: What are edge deployment and on-device inference constraints? How do you deploy LLMs on phones/IoT?

**A:** Edge deployment (phones, IoT devices, browsers) requires extreme optimization: models are 1GB-5GB (phone storage), inference happens without internet (completely on-device), latency budget is typically 1-2 seconds (user tolerance), compute is limited (mobile CPU, not GPU). Constraints: (1) model size (70B model is 140GB uncompressed, 35GB INT8, 17.5GB INT4—exceeds phone storage), (2) memory (iPhone has 6-8GB RAM, must fit model + activations), (3) latency (0.5-2 second budget for interactive response), (4) power (battery efficiency critical). Optimization approaches: (1) aggressive quantization (INT4, INT2, bit-packing reduce to 1-2GB), (2) knowledge distillation (train 3B model from 70B teacher, achieving reasonable quality), (3) structured pruning (remove 50%+ of layers), (4) attention optimizations (multi-query attention reduces KV cache), (5) operator fusion (combine multiple ops into single kernel, reducing overhead). Example: 7B model quantized to INT4 = 3.5GB (barely fits), distilled from 13B teacher, with pruned 20% of layers, achieves ~80% of original quality on summarization. Frameworks: (1) ONNX Runtime (CPU inference, iOS/Android support), (2) MediaPipe (Google's on-device ML framework), (3) Core ML (Apple), (4) TensorFlow Lite (general-purpose edge), (5) specialized solutions (llama.cpp for pure CPU, MLX for Apple Silicon). Practical deployment: (1) use smaller base models (Phi-3, Mistral-7B are better than Llama-70B for edge), (2) quantize aggressively (INT4 essential), (3) prune non-critical layers, (4) optimize operators for target hardware (ARM vs. Apple Silicon), (5) test on-device thoroughly (emulators don't accurately reflect real performance). Trade-offs: quality degradation (INT4 + distillation loses 10-20% capability), latency (still slow—token/sec on phone, not tokens/sec on GPU), battery drain (inference uses 50% battery for a few minutes of heavy use). Edge deployment is frontier for LLMs; current capabilities (3-7B model, low-quality output) are limited but improving rapidly.

### Q15: Describe best practices for LLM inference optimization. How do you profile bottlenecks and evaluate optimization tradeoffs?

**A:** Effective LLM inference optimization requires systematic approach. Step 1: Measure baseline. Profile actual workload: measure TTFT, ITL, throughput, latency percentiles (p50, p99, p999), memory usage. Understand bottleneck: is it memory-bandwidth (model is huge relative to compute) or compute-bound (GPU can't keep up)? Memory-bound: optimize memory access (quantization, KV-cache optimization, FlashAttention). Compute-bound: increase parallelism (batching, TP). Step 2: Target optimization. Prioritize based on pain points: if latency is too high, reduce batch size, use speculative decoding, optimize TTFT. If throughput is low, increase batch size, enable paged attention, use continuous batching. If cost is high, quantize, use model routing, use spot instances. Step 3: Implement incrementally. Start with low-effort, high-impact: enable paged attention (5% code, 2-3x throughput). Then quantization (INT8, small accuracy loss, 2x speedup). Then speculative decoding (complex, 2x speedup). Avoid over-optimizing (don't spend 3 months on 5% speedup if baseline is adequate). Step 4: Evaluate tradeoffs. Quality metrics (accuracy on benchmarks + downstream task metrics—summarization, QA, retrieval), latency (measure on representative workload, not synthetic), cost (compute + infrastructure). Pareto-optimal solutions: model A is faster but lower quality; model B is higher quality but slower—choose based on business priorities. Step 5: Monitor production. Measure real-world performance: actual latency on real requests (not synthetic), quality via user feedback, cost per request. Continuously optimize: if workload patterns change (more long requests), re-optimize. Best practices: (1) never optimize without measuring (intuition fails), (2) make one change at a time (understand impact), (3) evaluate on representative data (benchmarks ≠ production), (4) automate (continuous profiling, automated optimization), (5) document decisions (why was this optimization chosen, what tradeoffs were made). Common pitfalls: (1) over-optimizing for benchmark (MMLU score doesn't predict downstream quality), (2) ignoring latency tail (p99 matters more than average), (3) forgetting warm-up (JIT compilation, GPU cache warm-up affects latency), (4) ignoring data curation (better data beats better model). Systematic optimization can achieve 10-100x speedup or 5-10x cost reduction, making difference between viable and infeasible deployment.

## Interview Cheatsheet

**Key Terms:**
- **Quantization:** Reducing precision (FP32 → INT8 → INT4) to decrease memory and increase speed; tradeoff with accuracy.
- **KV-Cache:** Storing key-value embeddings from previous tokens; eliminates recomputation during decoding.
- **Paged Attention:** Managing KV-cache in fixed-size blocks, enabling higher batch sizes and memory efficiency.
- **Speculative Decoding:** Draft model generates K candidates; target model verifies all in one batch, parallelizing generation.
- **TTFT:** Time-to-First-Token; latency from request to first generated token (prefill-dominated).
- **ITL:** Inter-Token Latency; time between successive generated tokens (decoding-dominated).
- **FlashAttention:** Memory-efficient attention implementation using tiling and block-wise computation; 2-3x faster, 10x less memory.
- **Tensor Parallelism:** Splitting model parameters across GPUs horizontally; high communication, good for small GPU counts.
- **Pipeline Parallelism:** Splitting layers across GPUs vertically; low communication, scales to many GPUs.

**Rapid-Fire Q&A:**
- **Q:** Why is KV-cache more important than model size for inference memory? **A:** KV-cache grows with batch size and sequence length; for large batches or long sequences, KV-cache dominates memory, often exceeding weights.
- **Q:** Should you use INT8 or INT4 quantization? **A:** INT8 for most tasks (minimal quality loss, fast); INT4 only if memory is critical and you can validate quality loss is acceptable.
- **Q:** What's the main bottleneck in LLM inference? **A:** Memory bandwidth; GPUs have more bandwidth than needed to compute, but can't move data fast enough, limiting batch size and sequence length.
- **Q:** How does speculative decoding maintain quality if draft model is weak? **A:** Rejection sampling ensures output distribution matches target model exactly; weak draft model just has lower acceptance rate (less speedup), not lower quality.
- **Q:** When would you choose CPU inference over GPU? **A:** Only for extremely low-volume inference where GPU can't be amortized, or edge devices where GPU unavailable; CPU is 100x slower for LLMs.
