# Retrieval-Augmented Generation and Vector Databases

📺 **Video Lecture:** https://youtu.be/zw4WmjWg3LY


## Interview Anchor
- **RAG (Retrieval-Augmented Generation):** Augment LLM with retrieved relevant documents; improves factuality and enables external knowledge without retraining
- **Vector Database:** Specialized database for storing and searching embeddings via similarity; enables efficient semantic search at scale
- **Semantic Search:** Find documents similar to query via embedding similarity (cosine, dot product) rather than keyword matching

## Key Concepts Overview

Retrieval-Augmented Generation represents a paradigm shift in how language models handle external knowledge and factuality. Rather than encoding all knowledge in model parameters (which requires retraining), RAG retrieves relevant context at inference time and conditions generation on it. This approach decouples knowledge (in documents) from reasoning (in LLM), enabling: (1) Up-to-date information (documents change, model static). (2) Reduced hallucination (condition on facts). (3) Interpretability (cite sources). (4) Efficient scaling (add documents without retraining). Vector databases are the infrastructure enabling RAG—they efficiently find relevant documents for a query using semantic similarity. Understanding RAG and vector DBs is critical because: (1) Production systems widely use RAG (reducing hallucination is paramount). (2) Vector DBs are infrastructure—interviews test practical knowledge. (3) Chunking, embedding selection, indexing trade-offs are important engineering decisions. (4) Recent research shows RAG dramatically improves factuality and reduces hallucination. (5) Combining retrieval + generation requires understanding both components and their interaction.

### Q1: What is Retrieval-Augmented Generation (RAG)? Explain the architecture and benefits.

**A:** **RAG:** augment language model with retrieved external documents. Architecture: (1) **Query encoding:** encode user query to embedding. (2) **Retrieval:** search vector database for k most similar documents. (3) **Ranking/Re-ranking:** optionally re-rank retrieved documents (using cross-encoder for better relevance). (4) **Prompt construction:** concatenate query + top documents → prompt. (5) **Generation:** LLM generates answer conditioned on documents. (6) **Post-processing:** extract answer (optionally cite sources). Example: Q: "What is CRISPR?" → retrieve biology articles → generate answer grounded in documents. **Benefits:** (1) **Factuality:** condition on ground-truth documents (reduced hallucination). (2) **Up-to-date:** documents versioned (news, prices, events), no retraining. (3) **Transparency:** cite sources (why answer is correct). (4) **Efficiency:** don't encode all knowledge in model (save parameters). (5) **Flexibility:** swap documents without retraining. (6) **Scoping:** restrict to domain-specific documents (medical RAG only uses medical journals). **Limitations:** (1) **Retrieval errors:** if top-k documents don't contain answer, generation fails. (2) **Ranking errors:** relevant documents ranked low (curse of high dimensions). (3) **Context window:** can't retrieve all documents (fitting k documents in context trades recall for relevance). (4) **Computational cost:** retrieval + reranking + generation slower than direct generation. (5) **Quality depends on documents:** garbage documents → garbage answer (GIGO). **Empirical results:** RAG substantially reduces hallucination (20-40% improvement on factual QA benchmarks). Used in production: OpenAI's ChatGPT plugins use RAG, Anthropic Claude uses RAG, Google Bard uses retrieval. RAG is now standard for factual applications (Q&A, research, customer support).

**Interview Tip:** Explain the full pipeline end-to-end. Emphasize hallucination reduction (key motivation). Mention that retrieval is not perfect (failures cascade to generation). Show understanding of the tradeoff (more documents → better recall, worse latency).

### Q2: Explain chunking strategies for documents. How does chunk size affect retrieval?

**A:** **Chunking:** splitting documents into smaller pieces for embedding and retrieval. Strategies: (1) **Fixed-size chunks:** split into N-token chunks (e.g., 256 tokens). Simple, uniform. Disadvantage: may split sentences (loses context). (2) **Recursive chunking:** split by meaningful boundaries (sentences, paragraphs, sections) recursively. Example: first split by section, if section too large, split by paragraph, if paragraph too large, split by sentence. Preserves semantic units. (3) **Semantic chunking:** use embeddings to find natural breakpoints. Split when semantic similarity between tokens drops sharply. More sophisticated, but slower. (4) **Document-aware chunking:** respect document structure (don't split headers, tables, code blocks). Requires parsing. **Chunk size trade-off:** (1) **Small chunks (e.g., 128 tokens):** (a) More chunks → higher recall (more opportunities to find relevant info). (b) Cleaner context (less noise). (c) Slower search (more vectors). (d) Risk: loses context (a chunk without surrounding context may be ambiguous). (2) **Large chunks (e.g., 1024 tokens):** (a) Fewer chunks → lower recall (may miss relevant info). (b) More context per chunk (clearer meaning). (c) Faster search (fewer vectors). (d) Risk: includes irrelevant context (may confuse LLM or fill up context window). **Empirical findings:** 256-512 token chunks often optimal (balance recall and clarity). Depends on task: factual QA benefits from small chunks (specific facts), open-ended questions benefit from larger chunks (context). **Overlap:** chunks can overlap (e.g., 256-token chunks with 50-token overlap) to reduce boundary effects. **Metadata:** store chunk metadata (document ID, section title, page number) for citation and filtering. **Modern approach:** semantic chunking (split at semantic boundaries) + recursive fallback (if semantic split fail, use fixed-size). Tools: LangChain, LlamaIndex provide chunking utilities. Best practice: experiment—try 256/512/1024, measure retrieval quality (recall@k), pick best.

**Interview Tip:** Explain the recall-clarity tradeoff concretely. Mention that chunk size affects both retrieval and generation (too large consumes context window). Show you understand this is empirical—no one-size-fits-all answer.

### Q3: Explain embedding models for retrieval. What properties matter?

**A:** **Embedding models:** encode text (query or document) to vector. Quality determines retrieval performance. **Key properties:** (1) **Semantic understanding:** embedding should capture meaning. Example: "dog" and "puppy" should be close, "dog" and "car" far. Learned via contrastive objectives (SimCLR, CLIP). (2) **Dimensionality:** vector dimension (typically 384, 768, 1024). Higher dimension → more capacity but slower/more memory. (3) **Normalization:** embeddings should be normalized (unit norm). Enables cosine similarity = dot product. (4) **Alignment:** query and document embeddings should be comparable. Ideally: query and relevant document close in embedding space. Achieved via training on query-document pairs. **Popular models:** (1) **BERT-based:** Sentence-BERT (SBERT), all-MiniLM-L6-v2 (small, fast). 384-dim, lightweight. (2) **Dense Passage Retrievers (DPR):** trained on QA pairs, excellent for fact retrieval. 768-dim. (3) **E5 (Multilingual):** trained on diverse retrieval tasks, supports 100+ languages. (4) **CLIP:** image-text model, enables multi-modal RAG. (5) **OpenAI embeddings:** proprietary, high-quality, expensive. (6) **Sentence Transformers:** open-source family, various sizes/speeds. (7) **LLM-based embeddings:** use LLM to encode text (e.g., CohereEmbeddings, OpenAI 3-small). Higher quality but slower. **Trade-offs:** (1) **Speed vs Quality:** small models (all-MiniLM: 5ms/query) vs large (OpenAI: 50ms). Production systems choose based on QPS (queries per second) requirements. (2) **Domain vs General:** general embeddings (trained on broad data) vs domain-specific (trained on medical/legal documents). Domain-specific better if available. (3) **Multilingual vs English:** multilingual slower but enable cross-language retrieval. **Empirical results:** SBERT/E5 achieve ~90-95% retrieval accuracy on standard benchmarks (BEIR). Specialized models (DPR on SQuAD) exceed 99%. **Evaluation:** test on benchmark (BEIR—111 datasets across retrieval tasks), measure NDCG@10 or MRR. **Modern practice:** use SBERT or E5 as default (fast, effective, open-source). Fine-tune if custom domain and resources available. For production QA: use specialized model (DPR) if QA is primary use case.

**Interview Tip:** Discuss the alignment concept (query-document embeddings should match). Show understanding of speed-quality tradeoff. Mention that choice of embedding model dramatically affects RAG quality. Suggest fine-tuning on task-specific data (important for domains).

### Q4: Explain vector similarity measures: cosine, dot product, L2 distance. When use each?

**A:** **Similarity measures:** quantify how similar two embeddings are. **Cosine similarity:** `cos(a, b) = (a·b) / (||a|| ||b||)`. Ranges [-1, 1] (or [0, 1] for normalized embeddings). Invariant to magnitude; only cares about direction. **Dot product:** `a·b`. Ranges (-∞, ∞). Sensitive to magnitude. If embeddings normalized (||a|| = 1), dot product = cosine. **L2 distance:** `||a - b||_2 = sqrt(Σ(a_i - b_i)^2)`. Ranges [0, ∞). Euclidean distance. When use: (1) **Cosine similarity:** standard for embeddings (normalized, invariant to magnitude). Easy to interpret ([-1, 1] range). Compute efficient after normalization (becomes dot product). Used everywhere. (2) **Dot product:** equivalent to cosine for normalized embeddings; slightly faster (no normalization needed). Used in FAISS, Pinecone (internally uses dot product for speed). (3) **L2 distance:** less common for embeddings (sensitive to magnitude). More interpretable as geometric distance. Useful if you need actual distances (e.g., for clustering radius). **Implementation:** (1) In FAISS, vector databases: cosine stored as normalized vectors, similarity computed via dot product. (2) In dense retrieval: compute query-document dot products (via matrix multiply), top-k by score. (3) **Inverse relationship:** large cosine similarity ≈ small L2 distance (opposite directions for ranking). **Example:** (a) Query embedding: [0.9, 0.4], (b) Doc A: [0.85, 0.42], (c) Doc B: [0.1, 0.99]. Cosine(Q, A) ≈ 0.999 (very similar), Cosine(Q, B) ≈ 0.46 (dissimilar). L2(Q, A) ≈ 0.08 (close), L2(Q, B) ≈ 1.26 (far). Same ranking. **Modern practice:** cosine standard for dense retrieval. Ensure embeddings normalized (||v|| = 1) for efficiency. Some systems don't normalize and use dot product (faster), but assumes embeddings roughly normalized (which is typical for trained models).

**Interview Tip:** Explain cosine as standard for embeddings (invariant to magnitude). Show that cosine = dot product for normalized embeddings (optimization). Mention that choice matters less than ensuring consistency (normalize or not, but be consistent).

### Q5: Explain vector database indexing: HNSW, IVF, PQ. Trade-offs?

**A:** Vector databases use specialized indexes for efficient similarity search. Naive approach: brute-force search (compute similarity to all vectors, O(n) slow). Better: index that groups similar vectors, search only relevant groups. **HNSW (Hierarchical Navigable Small Worlds):** (1) **Idea:** hierarchical graph where vectors are nodes, edges connect nearby vectors. Starts at top layer (sparse), navigates to relevant region, then descends to find nearest neighbors. (2) **Search:** O(log n) hops to find region, O(k) to find top-k neighbors. Fast in practice (sublinear). (3) **Construction:** O(n log n) time, requires tuning (M = # neighbors, ef_construction = size of candidate set). (4) **Memory:** stores graph structure (~8-16 bytes per edge per vector), significant overhead. (5) **Advantages:** fast search, good for high-dimensional, no need to specify k beforehand. (6) **Disadvantages:** higher memory, construction slower than IVF. **IVF (Inverted File Index):** (1) **Idea:** cluster vectors (e.g., K-means), for each cluster store list of vectors. Search: find closest clusters, search within clusters. (2) **Search:** O(k) to find top clusters, O(nprobe) to search. nprobe is hyperparameter (larger = slower, more accurate). (3) **Construction:** O(nk) for clustering (fast). (4) **Memory:** minimal overhead (just cluster centroids). (5) **Advantages:** simple, fast construction, tunable (adjust nprobe for accuracy-speed tradeoff). (6) **Disadvantages:** performance degrades in high-dimensional (curse of dimensionality). **PQ (Product Quantization):** (1) **Idea:** split vector into m subspaces, quantize each subspace to fewer bits (e.g., 8 bits = 256 values). Store quantized codes (compact, ~1-4 bytes per vector). (2) **Search:** compute distance from query to all cluster centers (in each subspace), fast approximation. (3) **Construction:** train quantizer (k-means in each subspace), O(n) to encode. (4) **Memory:** very efficient (1-4 bytes per vector vs 4-16 bytes original float32). (5) **Advantages:** massive memory reduction (100x), fast search. (6) **Disadvantages:** lossy compression (slight accuracy loss), reconstruction not exact. **Combinations:** modern databases use hybrids. Example: IVF + PQ (coarse quantization via IVF, fine quantization via PQ). HNSW + PQ possible but less common. **Trade-off table:** HNSW (fast, high memory), IVF (balanced), PQ (compact, lossy). **Empirical results:** HNSW fastest (5-10ms for 1M vectors, 1M queries), IVF balanced (10-50ms depending on nprobe), PQ compact (small memory, similar speed to IVF). **Production systems:** Pinecone uses HNSW at scale, Weaviate uses HNSW, Milvus supports all (IVF, HNSW, PQ). Choice depends: recall requirements (HNSW best), memory constraints (PQ best), construction time (IVF best).

**Interview Tip:** Explain the search complexity (HNSW O(log n), IVF O(nprobe)) vs brute-force O(n). Discuss the memory-accuracy tradeoff (HNSW high memory, PQ low). Show you understand that choice affects latency and cost at scale (1M+ vectors).

### Q6: Explain hybrid search: combining dense retrieval and sparse (BM25) retrieval.

**A:** **Dense retrieval (semantic):** embedding-based, captures meaning. Query and documents encoded to vectors, ranked by similarity. Advantages: semantic understanding (synonyms matched), few documents, real-time. Disadvantages: requires embedding model, can't handle exact phrases, slower than sparse. **Sparse retrieval (BM25):** keyword-based, exact matching. Query terms matched to document terms, ranked by term frequency/inverse frequency. Advantages: interpretable (which terms matched), fast, handles exact phrases. Disadvantages: no semantic understanding (dog ≠ puppy), brittleness (typos fail). **Hybrid:** combine both. Architecture: (1) Retrieve top-k_dense from dense index. (2) Retrieve top-k_sparse from BM25 index. (3) Merge and rank. Merging strategies: (a) Reciprocal Rank Fusion (RRF): `score = Σ 1 / (k + rank_i)` for each index. Combines ranks without explicit scores. (b) Normalized scores: normalize dense scores (0-1) and sparse scores (0-1), weighted combination: `score = α * dense_score + (1-α) * sparse_score`. (c) Learning-to-rank: train supervised model to combine signals. **Why effective:** (1) Dense catches semantic matches (semantically similar documents ranked high). (2) Sparse catches exact matches (keywords boosted). (3) Complementary: a document that is neither semantically similar nor exact match may be caught by one but not other. (4) Robustness: if dense embedding fails (rare documents), sparse provides fallback. **Empirical results:** hybrid search outperforms both alone (5-15% improvement on BEIR). Dense + sparse ≈ dense alone + learning-to-rank (if trained). **Trade-off:** hybrid slower (search both indexes), but accuracy improvement often worth it. **Modern systems:** production RAG systems increasingly use hybrid. Elasticsearch supports hybrid (BM25 + dense). Pinecone, Weaviate enable hybrid setup (external BM25 index + vector index). **Implementation:** (1) Maintain two indexes (dense vector DB, sparse BM25 engine). (2) At query time, query both in parallel. (3) Merge top-k results (merge duplicates, re-rank). (4) Can be slow; async retrieval or parallel queries help. Alternative: reranking only (retrieve with sparse, rerank with dense cross-encoder). Simpler, lower latency.

**Interview Tip:** Explain complementarity (dense semantic, sparse exact). Discuss why both matter (examples: synonyms require dense, exact phrases require sparse). Show you understand modern production setups use hybrid. Mention RRF as simple merging strategy (no need to tune α weights).

### Q7: Explain re-ranking in RAG. Why is it important?

**A:** **Re-ranking:** retrieve large set (e.g., 100 documents) with fast retrieval, then re-rank top subset with expensive but accurate model. Motivation: trade-off speed and accuracy. Fast retrieval (dense embedding) may miss relevant documents or rank them low (curse of dimensionality). Expensive re-ranker (cross-encoder) provides more accurate scoring. **Architecture:** (1) **Retriever:** encode query and documents, rank by embedding similarity. Fast (single forward pass), supports large-scale. (2) **Re-ranker:** cross-encoder takes (query, document) pair as input, outputs relevance score. Single forward pass per pair, expensive but accurate. Example: (retrieve 100 documents in 5ms, re-rank top 50 in 50ms). **Cross-encoder:** neural network scoring (query, doc) pair. Architecture: concatenate query and doc, pass through encoder (e.g., SBERT or BERT), output single relevance score. Loss: train on ranking objective (predict which doc more relevant). **Why effective:** (1) **Joint interaction:** encoder sees query-doc interaction (not independent), better ranking. (2) **Large context:** can condition on full document (not just compressed embedding). (3) **Learned ranking:** trained end-to-end on ranking task (vs. hand-crafted BM25). Empirically: cross-encoder NDCG@10 often 5-10% better than biencoder (retriever). **Cost-benefit:** re-ranking adds latency (~1-10ms per document), but improves accuracy. Production trade-off: rank top-k only (e.g., top 100 or 1000 from retriever), re-rank top-m (e.g., top 20). **Models:** (1) **mBERT (multilingual BERT):** small, fast, open-source. ~50ms for 1 query vs 100 docs. (2) **ms-marco-MiniLM-L-12-v2:** trained on ranking data, very good. (3) **Cross-encoders from HuggingFace:** domain-specific (legal, medical). (4) **LLM-based:** use ChatGPT or Claude to rerank. Expensive but can justify for critical queries. **Implementation:** (1) Retrieve k documents (e.g., 100). (2) Batch re-rank top m (e.g., 20). (3) Return top-n (e.g., 5) to generation. Pipeline reduces latency (re-rank only top-20, not all 100). **Modern practice:** re-ranking standard in production RAG systems (Vespa, Elasticsearch with plugins, custom implementations). Especially important if retriever is weak (SBERT only) or dataset large (high noise in top-100). Research ongoing: how to make re-rankers faster (distillation, quantization, approximation).

**Interview Tip:** Explain the retriever-re-ranker pipeline (fast coarse, accurate fine). Show the latency-accuracy tradeoff (acceptable 50ms extra for 10% accuracy improvement). Mention that re-ranking is critical when retriever weak or false negatives costly.

### Q8: Explain metadata filtering in vector databases. How does it improve retrieval?

**A:** **Metadata:** auxiliary information about documents (date, author, category, source, etc.). Stored alongside vectors, enables filtering. **Use cases:** (1) **Temporal filtering:** "documents from last 30 days" (freshness). (2) **Source filtering:** "only peer-reviewed papers" (quality). (3) **Category filtering:** "medical documents only" (domain). (4) **Author/permission:** "documents user has access to" (privacy). **Architecture:** (1) **Storage:** each vector has metadata (key-value pairs). Example: vector + {source: "wikipedia", date: "2024", category: "science"}. (2) **Filtering:** at query time, filter vectors by metadata constraints. Example: `source == "wikipedia" AND date >= "2023"`. (3) **Search:** search only filtered vectors. **Implementation:** (1) **Pre-filter:** apply SQL-like filter before vector search. Query: "find vectors where source='Wikipedia'" (SQL predicate) → retrieve subset → vector search on subset. Fast (small subset), but may miss results if filter too restrictive. (2) **Post-filter:** vector search on all vectors → filter results by metadata. Slower (search all), but doesn't miss results. (3) **Index-aware:** some databases (Weaviate) combine metadata indexing and vector indexing (HNSW can be partitioned by metadata). Hybrid: fast pre-filter + accurate search. **Trade-off:** filtering reduces search space (faster), but if filter removes relevant vectors, recall drops. Example: search "machine learning" in last 7 days → may miss relevant older papers. Balance recall vs precision. **Implementation details:** (1) Metadata indexing: index metadata (date, category) for fast lookups (B-tree, hash table). (2) Selectivity: if filter matches 90% of vectors, benefit small. If <10%, significant speedup. (3) Cardinality: high-cardinality metadata (dates, specific users) indexable; low-cardinality (category: 5 types) less benefit. **Empirical results:** filtering adds ~1-5ms (small overhead for indexing), can reduce search space 10-100x (significant speedup). **Modern systems:** Pinecone, Weaviate, Milvus all support metadata filtering. **Example use case:** multi-tenant RAG. Each tenant has documents. Query from tenant A → pre-filter to tenant A's documents. Prevents data leakage.

**Interview Tip:** Explain the pre-filter vs post-filter tradeoff (speed vs recall). Discuss real-world use case (multi-tenant filtering for security). Mention that good metadata design (indexed, low-cardinality) is important for performance.

### Q9: Explain RAG evaluation metrics: faithfulness, relevance, context recall.

**A:** **Evaluation challenge:** RAG combines retrieval (retrieve relevant docs) and generation (answer from docs). Metrics must assess both components. **Key metrics:** (1) **Context Recall:** does retrieved context contain answer? Measure: does top-k retrieved documents contain sufficient information to answer the question? Compute: label which documents are relevant (human annotation or model-based proxy), compute recall (fraction of relevant docs in top-k). Formula: `recall@k = |relevant_docs_in_topk| / |total_relevant_docs|`. Typical thresholds: recall@5 or recall@10. (2) **Relevance (Retrieval Precision):** are retrieved documents relevant to query? Measure: manually judge top-k documents as relevant/not. Compute: `precision@k = |relevant_in_topk| / k`. NDCG@k combines ranking (higher rank = more credit). **Faithfulness:** is generated answer grounded in context? Two approaches: (a) **Entailment-based:** use NLI (natural language inference) model to check if answer entailed by context. Answer faithful if `NLI(context, answer) = ENTAIL`. (b) **Factual consistency:** check facts in answer against context (requires fact extraction). Metrics: `faithfulness = |facts_in_answer_supported_by_context| / |total_facts_in_answer|`. (3) **Answer Relevance:** does answer address the question? Measure: LLM-based (ChatGPT judges if answer relevant) or human. (4) **Token Overlap (BLEU/ROUGE):** simple proxy (answer similar to expected). Problem: multiple valid answers, penalizes paraphrases. **Combined metrics:** (a) **Context Recall + Faithfulness:** answer can only be faithful if context allows. If context recall low, faithfulness naturally low (no info in context). (b) **End-to-end accuracy:** whether final answer correct (ground truth). Hardest to get, but most meaningful. **Evaluation pipeline:** (1) Create test set (queries with gold answers + relevant documents). (2) Run RAG system (retrieve documents, generate answer). (3) Compute metrics: context recall (are docs relevant?), faithfulness (is answer grounded?), answer relevance (does answer address query?), end-to-end accuracy. (4) Diagnose failures: low context recall → retriever problem. Low faithfulness → generator problem. (5) Iterate: improve retriever (better embeddings, re-ranking) or generator (better prompts, fine-tuning). **Empirical observations:** (1) Context recall often bottleneck—if relevant docs not retrieved, generation fails. (2) Faithfulness improves dramatically with RAG (20-40% hallucination reduction). (3) Answer relevance sometimes drops slightly (focus on context may miss broader context). **Tools:** RAGAS (RAG assessment library) provides implementations of these metrics.

**Interview Tip:** Emphasize the two-stage evaluation (retrieval accuracy + generation quality). Explain why context recall matters (answer limited by what's retrieved). Mention the diagnosis workflow (identify bottleneck, improve). Discuss the tradeoff (perfect context recall → long documents → fills context window).

### Q10: Explain advanced RAG patterns: self-RAG, corrective RAG, agentic RAG.

**A:** Modern RAG goes beyond retrieve-then-generate; systems now decide when to retrieve, correct answers, etc. **Self-RAG:** model learns when retrieval is needed. Training: (1) Add special tokens (e.g., [RETRIEVE], [SKIP]) to model vocabulary. (2) Fine-tune model to predict when to retrieve (before generating). Decision: if query needs external info, output [RETRIEVE], trigger retrieval and generation. (3) Combine LM + retrieval: model can interleave generation and retrieval steps. Advantage: saves compute (skip retrieval when not needed). Empirically: self-RAG 3-5% better than always-retrieve baseline. **Corrective RAG:** system checks answer, corrects if wrong. Pipeline: (1) Generate answer (from context or without). (2) Fact-check: verify facts against context (NLI or entailment). (3) If fails checks: retrieve more documents, regenerate. (4) Repeat until pass checks or max iterations. Advantage: self-correcting (improves answer quality). Empirically: 5-10% improvement on factual QA. **Agentic RAG:** system acts autonomously, deciding retrieval strategy. Example: (1) User asks complex question requiring multiple facts. (2) Agent breaks down: sub-queries. (3) For each sub-query: retrieve documents, generate answer, check. (4) Combine answers. (5) Return final answer + reasoning steps. Advantage: handles complex queries requiring reasoning + retrieval. Similar to ReAct prompting (reasoning + acting). Empirically: better on multi-hop reasoning (requires multiple retrieval steps). **Examples:** (1) **Self-RAG (Asai et al.):** fine-tuned LLaMA with [RETRIEVE] tokens, outperforms always-retrieve baselines. (2) **Corrective RAG:** Azure Cognitive Search uses similar idea (fact-check, re-retrieve if needed). (3) **Agentic RAG:** LangChain agents use this (define tools = retrieve, use LLM to decide which tools). **Implementation:** (1) Self-RAG: requires model fine-tuning. (2) Corrective: wrap generation + fact-check + conditional retrieval. (3) Agentic: use tool-calling (LLM can invoke functions), define retrieval as tool. **Tradeoff:** more sophisticated → higher latency (more steps), but better quality. Production systems choose based on latency budget and accuracy requirements. **Modern trend:** from static RAG to adaptive/agentic RAG. Research active; new papers monthly on variants.

**Interview Tip:** Explain the progression (fixed RAG → self-RAG → corrective → agentic). Show you understand each adds intelligence/automation. Discuss use cases (self-RAG for efficiency, corrective for accuracy, agentic for complex reasoning). Mention that sophisticated approaches have latency cost.

### Q11: Explain context window management in RAG. How to handle long documents?

**A:** **Context window limit:** LLMs have max sequence length (GPT-4: 8K-128K tokens, LLaMA: 4K-100K). RAG concatenates query + documents + answer in one prompt. Context window fills quickly, limiting # documents or document length. **Challenge:** (1) k documents retrieved, concatenate to prompt. If documents large, fit only few documents before hitting context limit. (2) As k increases, latency increases (process longer prompt). (3) Quality may degrade: long context → model attends to beginning/end more ("lost in the middle" problem). **Strategies:** (1) **Chunk documents:** split long documents into smaller chunks (256-512 tokens), retrieve chunk-level (not document-level). Tradeoff: more chunks → more retrieval candidates → slower, but better granularity. (2) **Top-k only:** retrieve only top-k documents (e.g., k=5) to fit in context. Tradeoff: fewer documents → lower recall, but faster. (3) **Hierarchical retrieval:** retrieve coarse (document-level), then fine-grained (chunk-level). Example: retrieve 5 relevant documents, then within each, retrieve most relevant chunk. (4) **Summarization:** summarize each document, include summary + key quotes. Reduces tokens while preserving info. Tradeoff: summary may miss details. (5) **Long-context models:** use GPT-4-Turbo (128K context) to fit more documents. Cost trade-off (more expensive). (6) **Compression:** compress context using extractive (select key sentences) or abstractive (generate summary) compression. (7) **Prompt optimization:** remove redundant/irrelevant context (learned via optimization). (8) **Document ranking before context:** rank documents by relevance, include top-k in context. Ensures most relevant info first (helps with lost-in-the-middle). **Implementation:** (1) After retrieval, rank retrieved documents (via cross-encoder or simple heuristic). (2) Select top-k to fit in context window (compute tokens, ensure under limit). (3) Construct prompt: "Answer the question based on provided context. Context: [doc1] [doc2] ... [docN]. Question: [q]". (4) Generate. **Modern practice:** chunk-level retrieval + top-5/10 + long-context model (if available). Many systems use ~5K tokens for context, leaving room for answer generation. **Empirical results:** increasing context from 2K to 5K helps (5-10% improvement), beyond 5K diminishing returns (lost-in-the-middle effects). **Trade-off table:** Small k (fast, lower accuracy), large k (slow, higher accuracy). Typical sweet spot: k=5-10, chunk_size=256-512.

**Interview Tip:** Explain that context window is precious resource (trade-off between # documents and token budget). Discuss chunking as way to increase retrieval granularity without increasing document size. Mention lost-in-the-middle (modern problem, counterintuitive to naive RAG).

### Q12: Explain multi-modal RAG: combining text, images, tables. How does it change retrieval?

**A:** **Multi-modal RAG:** retrieve and generate from documents containing text, images, tables, etc. Motivation: real documents (PDFs, webpages) are multi-modal. Text-only RAG ignores images/tables (information loss). **Challenges:** (1) **Encoding:** need embeddings for different modalities (text embeddings differ from image embeddings). (2) **Alignment:** query may ask for text answer about images (need alignment between modalities). (3) **Retrieval:** search across mixed types (retrieve text documents and images). (4) **Generation:** generate text answer grounded in image/table. **Approaches:** (1) **Unified embedding space:** project all modalities to same vector space. Example: CLIP (image-text model) projects image and text to 512-dim space (aligned). Search: embed query (text) and documents (text/images), similarity in unified space. Advantage: simple, end-to-end. Disadvantage: loses modality-specific info. (2) **Modality-specific embeddings:** maintain separate embeddings for each modality. Retrieve with modality-specific metric. Advantage: specialized (better accuracy), Disadvantage: need multiple models, complex. (3) **Document structure:** preserve document structure (image in context of surrounding text). Example: "Figure 3 shows ... [image] ... which demonstrates ...". Retrieve and pass structure to LLM. (4) **OCR for tables:** convert tables to structured text (SQL, JSON), embed and retrieve. **Implementation:** (1) Parse documents: extract text, images, tables. (2) Embed: text → text embeddings, images → CLIP embeddings (or GPT-4V embeddings), tables → structured representations (SQL or semantic vectors). (3) Retrieve: embed query, search across all modalities (hybrid index or unified space). (4) Rank: re-rank to ensure best matches (may need multi-modal cross-encoder). (5) Generate: concatenate multimodal context (text + images + table descriptions) → LLM generates answer. Alternative: use multimodal LLM (GPT-4V, Claude 3 Vision) to generate from images directly. **Examples:** (1) **CLIP-based RAG:** embed documents (images + text descriptions) to CLIP space, query similarly. (2) **Vision RAG:** parse PDFs with images, use vision transformer to embed images, retrieve similar images for query. (3) **Table QA:** retrieve relevant tables (via content + metadata), generate answer from table data. **Empirical results:** multi-modal RAG improves on visual QA (VQA) benchmarks 20-30% over text-only. **Modern tools:** LangChain supports multi-modal (via APIs like GPT-4V for image analysis, vector DBs for mixed embeddings). **Trade-off:** complexity (handle multiple modalities) vs coverage (include all information types). Most production systems start with text, add images/tables as needed.

**Interview Tip:** Explain the unified embedding space concept (project image/text to same space like CLIP). Discuss the modality-specific tradeoff (generality vs specialization). Mention that GPT-4V enables simpler approach (encode images to LLM directly, skip explicit image retrieval).

### Q13: Explain knowledge graphs in RAG. How do they improve retrieval and reasoning?

**A:** **Knowledge graph (KG):** structured representation of facts: entities (nodes) and relationships (edges). Example: (Entity: OpenAI, Relation: founded_by, Entity: Sam Altman). Compared to unstructured documents: KG explicit, structured, queryable via graph algorithms. **Knowledge graph + RAG:** combine structured reasoning (KG) + semantic reasoning (embeddings). Architecture: (1) **Knowledge graph construction:** parse documents, extract entities and relationships (NER + relation extraction). Build KG: nodes = entities, edges = relationships. (2) **Encoding:** embed entities and relations to vectors. Similar entities get similar embeddings. (3) **Query expansion:** given query, traverse KG to find related entities. Example: "Who founded OpenAI?" → query node [OpenAI] → traverse edge [founded_by] → retrieve related entity [Sam Altman]. (4) **Retrieval:** retrieve documents mentioning traversed entities. (5) **Generation:** pass retrieved documents + KG context → LLM generates answer. **Benefits:** (1) **Explicit reasoning:** KG enables graph traversal (multi-hop reasoning without document retrieval). Example: "Who did Sam Altman work with after OpenAI?" → traverse founded_by (reverse) → find entities founded by Sam → traverse work_with edges. (2) **Structured knowledge:** useful for reasoning over facts (dates, relationships). (3) **Disambiguation:** if "OpenAI" mentioned in many contexts, KG links to single entity (no ambiguity in vector retrieval). (4) **Inference:** graph algorithms (shortest path, clustering) reveal new knowledge (not explicit in documents). **Challenges:** (1) **Construction:** building KG from documents is hard (NER/RE are imperfect). Manual construction expensive. (2) **Maintenance:** KG must be updated when facts change. (3) **Coverage:** KG may be incomplete (missing relationships). (4) **Complexity:** adds engineering overhead. **Modern approaches:** (1) **Hybrid:** combine unstructured retrieval (semantic search) + structured reasoning (KG traversal). If semantic retrieval fails, KG provides fallback. (2) **LLM-based construction:** use LLM to extract entities and relationships (more accurate than traditional NER/RE). (3) **Vector KG:** embed entities, use embeddings for similarity-based KG search (blend structured + unstructured). **Examples:** (1) **Biomedical KGs:** UMLS (medical ontology), use for medical RAG. (2) **Knowledge bases:** Wikidata, Freebase. (3) **Enterprise KGs:** company-specific (customers, products, relationships). **Empirical results:** KG improves on multi-hop reasoning (15-30% improvement on datasets requiring multiple reasoning steps). Less help on single-fact retrieval (semantic search sufficient). **Production use:** knowledge graphs used in enterprise RAG, biomedical AI, recommendation systems. But complex; most simple RAG systems skip KGs. **Research trend:** combining neural + symbolic reasoning (neural embeddings + KG structure) is active research area.

**Interview Tip:** Explain KG as explicit structure vs unstructured documents. Discuss multi-hop reasoning via graph traversal (KG advantage). Mention that KG construction from documents is hard (NER/RE imperfect), limiting adoption. Show understanding that KG valuable for domains with structured knowledge (medical, enterprise) but overkill for simple retrieval.

### Q14: Explain document parsing and preparation for RAG. Why is it important?

**A:** **Document parsing:** extracting text, structure, metadata from documents (PDF, Word, HTML, etc.). Quality affects downstream retrieval and generation. **Challenges:** (1) **Format diversity:** PDFs render text (no underlying structure), Word has markup, HTML has semantic tags. Parsing must handle all. (2) **Layout:** documents have headers, footers, multi-column layouts, images, tables. Parsing must preserve structure. (3) **Quality:** bad parsing creates noise (garbage in, garbage out). **Parsing approaches:** (1) **Text extraction:** use PDF libraries (PyPDF2, pdfplumber) to extract text. Problem: loses structure (doesn't know header vs body). (2) **OCR (Optical Character Recognition):** for scanned PDFs, convert image to text. Problem: errors, slow. (3) **Layout-aware parsing:** use document understanding models (LayoutLM, Nougat) to understand structure (detect headers, tables, sections). More accurate but slower. (4) **Semantic parsing:** extract semantic elements (title, abstract, sections). Useful for academic papers. **Preparation steps:** (1) **Parsing:** extract text from documents. (2) **Cleaning:** remove boilerplate (headers, footers, page numbers), normalize whitespace. (3) **Structure preservation:** keep section headers, bullet points, as they provide context. (4) **Chunking:** split into manageable pieces. (5) **Metadata extraction:** extract date, author, source, section title, for metadata filtering. (6) **Quality checks:** verify text makes sense (not corrupted), not mostly boilerplate. **Importance:** (1) **Retrieval quality:** good parsing → clean text → better embeddings → better retrieval. Bad parsing → noise → poor retrieval. (2) **Generation quality:** if source documents corrupt, generated answers reflect corruption (hallucination or incoherence). (3) **Citation accuracy:** if chunks not properly delimited, citations may be wrong (cite wrong section). **Tools:** (1) **Unstructured.io:** general document parsing (PDF, Word, HTML, images). (2) **LlamaIndex:** Document Loaders for many formats. (3) **PyPDF2, pdfplumber:** low-level PDF parsing. (4) **LayoutLM:** deep learning for layout-aware parsing. (5) **Cloud services:** Azure Form Recognizer, Google Document AI (paid, high accuracy). **Empirical observations:** good parsing 10-20% improvement in retrieval/generation quality vs naive parsing. **Best practice:** start with simple parsing, evaluate quality (spot-check samples), upgrade parsing model if needed. Don't assume parsing is "solved"—it's often a bottleneck.

**Interview Tip:** Explain that parsing is often overlooked but critical ("garbage in, garbage out"). Mention specific challenges (PDFs unstructured, OCR needed for scans). Discuss that good parsing is investment that pays off in downstream quality. Show you understand the full pipeline (parse → chunk → embed → retrieve → generate).

### Q15: Discuss challenges and future directions in RAG and vector databases.

**A:** **Current challenges:** (1) **Retrieval errors compound:** if retrieval fails, generation fails (even best LLM can't invent answers from wrong context). Improving retrieval robustness is critical. (2) **Lost-in-the-middle:** models attend to beginning/end of context, miss middle (even with long-context models). Solutions: hierarchical context, document ranking before context, compress less-relevant docs. (3) **Embedding gap:** embeddings may not capture all semantics (synonymy, negation, reasoning). Fine-tuning embeddings on task data helps but expensive. (4) **Scalability:** as document corpus grows (100M+), retrieval slows (indexing overhead, search time). Trade-off: accuracy vs speed. (5) **Multi-modality:** combining text/images/tables complex; most systems text-only. (6) **Real-time updates:** if documents change frequently (news, prices), indexing must be fast. Some vector DBs support incremental updates. (7) **Privacy:** documents may contain sensitive info; can't send to cloud APIs for embedding/generation. On-device solutions emerging (small models, local vector DB). **Future directions:** (1) **Better embeddings:** domain-adapted embeddings (fine-tune on task), sparse embeddings (combine dense + sparse signals), learned-to-rank embeddings (optimize for downstream task). (2) **Efficient retrieval:** approximate nearest neighbor that's faster + more accurate, adaptive retrieval (decide # documents dynamically). (3) **Integration with LLMs:** end-to-end training of retriever + generator (joint optimization). Currently separate. (4) **Structured reasoning:** combine KGs + neural retrieval more deeply (hybrid reasoning). (5) **Continual learning:** RAG systems that adapt as new documents added (avoid reindexing large corpus). (6) **Multimodal:** better multi-modal embeddings, video understanding, 3D object retrieval. (7) **Verifiable RAG:** generate not only answer but proof (retrievable documents that support answer). Currently no formal guarantees. (8) **Efficiency:** faster/smaller embeddings, quantized vector DBs, edge deployment. **Emerging paradigms:** (1) **In-context learning in retrieval:** instead of fixed retriever, use LLM to decide what to retrieve (agentic). (2) **Fusion of retrieval-augmented and generation-augmented:** combine with language model decoding (beam search over retrieval + generation). (3) **Retrieval for reasoning:** use retrieval not just for facts, but for reasoning steps (retrieve example reasoning paths, follow similar reasoning). **Research trends:** major conferences (ACL, EMNLP, ICLR) have RAG sessions. Active areas: improving retrieval robustness, scalability, privacy-preserving RAG, multi-modal RAG, integrated end-to-end systems.

**Interview Tip:** Discuss lost-in-the-middle as concrete problem (even with long-context, models struggle). Show awareness of the retrieval-generation pipeline bottlenecks (retrieval errors → cascade). Mention emerging directions (structured reasoning, continual learning) without claiming to be expert. Show that RAG is active research area, not solved. Discuss practical production challenges (efficiency, privacy, scale).

## Interview Cheatsheet

**Key Terms:**
- **RAG (Retrieval-Augmented Generation):** Augment LLM with retrieved documents; improves factuality, enables external knowledge, cites sources
- **Chunking:** Splitting documents into pieces; trade-off: small chunks (high recall), large chunks (high context)
- **Vector Database:** Specialized database for embedding storage/search; enables semantic search via similarity (cosine, dot product)
- **HNSW:** Hierarchical navigable small world index; O(log n) search, high memory, fast; standard in production
- **IVF:** Inverted file index via clustering; balanced speed/memory, tunable recall-speed via nprobe
- **PQ (Product Quantization):** Compress vectors to 1-4 bytes via quantization; efficient but lossy
- **Hybrid Search:** Combine dense (semantic, embeddings) + sparse (keyword, BM25); better than either alone
- **Re-ranking:** Expensive but accurate second-pass ranking; retriever (fast) + re-ranker (accurate) pipeline
- **Cross-Encoder:** Neural model scoring (query, document) pair; more accurate than bi-encoder but slower
- **Metadata Filtering:** Pre-filter vectors by attributes (date, category, source) before search; reduces search space
- **Context Recall:** Metric: do retrieved documents contain info needed to answer question?
- **Faithfulness:** Metric: is generated answer grounded in retrieved context? Reduce hallucination.
- **Self-RAG:** Model learns when to retrieve via special tokens; saves compute by skipping unnecessary retrieval
- **Corrective RAG:** Verify answer, re-retrieve if needed; self-correcting loop improves quality
- **Agentic RAG:** System acts autonomously, decides retrieval strategy; handles complex multi-hop queries

**Rapid-Fire Q&A:**
- **Q: RAG goal?** **A:** Reduce hallucination, cite sources, enable external knowledge without retraining
- **Q: Chunk size tradeoff?** **A:** Small: high recall, less context; large: low recall, more context. Optimal: 256-512 tokens
- **Q: Embedding model choice?** **A:** SBERT (fast baseline), DPR (QA-specialized), domain-specific if available. Test on BEIR
- **Q: Cosine vs L2 distance?** **A:** Cosine standard for embeddings (direction-only); L2 sensitive to magnitude. Both work if normalized
- **Q: Vector DB index choice?** **A:** HNSW (fast, high memory), IVF (balanced), PQ (compact). Production typically HNSW
- **Q: Why re-ranking?** **A:** Fast retriever (embedding) may rank poorly; expensive re-ranker (cross-encoder) refines top-k
- **Q: Hybrid search benefit?** **A:** Dense catches semantic, sparse catches exact matches. Complementary signals, 5-15% improvement
- **Q: Metadata filtering?** **A:** Filter vectors by attributes before search; enables multi-tenant isolation, temporal filtering, access control
- **Q: Context recall metric?** **A:** Fraction of relevant documents in top-k retrieved. Answers limited by what's retrieved
- **Q: Faithfulness metric?** **A:** Is answer entailed by context? NLI-based or fact-checking; should be >80% in good RAG
- **Q: Self-RAG innovation?** **A:** Model learns when to retrieve; saves compute by skipping unnecessary steps
- **Q: Lost-in-the-middle problem?** **A:** Even long-context models attend to beginning/end, miss middle. Solution: rank docs before context
